## Загальні вимоги до кожної статті

Щоб зберегти преміальний стиль `kostyl.dev`, кожна стаття повинна містити такі елементи:

1. **Вступ з проблематикою (Hook):** Замість сухої теорії починати з життєвої проблеми (наприклад: чому тисячі дрібних об'єктів «зжирають» гігабайти оперативної пам'яті в сервісі).
2. **Візуалізація структури:** Схеми PlantUML для ілюстрації зв'язків між об'єктами, класів-міксинів або алгоритму MRO.
3. **Механіка пам'яті та CPython Internals:** Пояснення, що відбувається під капотом (робота `__dict__`, вказівники на типи, лінеаризація класів).
4. **Конструктори Docus:**
    - `::debugger-view` або `::memory-view` для демонстрації стану об'єктів.
    - `::field-group` та `::field` для опису параметрів магічних методів або API.
    - `::terminal-preview` для виведення результатів тестування чи профайлінгу.
    - `::tip` / `::warning` / `::important` для виділення критичних нюансів та підводних каменів.
5. **Практичний блок (3 рівні складності):**
    - _Рівень 1 (Базовий):_ Закріплення синтаксису.
    - _Рівень 2 (Середній):_ Реальний production-сценарій.
    - _Рівень 3 (Advanced):_ Оптимізація продуктивності, безпека типів або написання міні-фреймворку.

### Модуль 1: Основи системи типів та Інкапсуляція

#### Стаття 1: Класи та Об'єкти: під капотом CPython

- **Файл:** `01.classes-objects.md`
- **Орієнтовний обсяг:** 3 000–4 000 слів
- **Основні теми:**
    - Оголошення класу та створення екземплярів. Що таке клас насправді в Python (об'єкт типу `type`).
    - Метод `__init__` проти `__new__`: хто насправді створює об'єкт, а хто його ініціалізує. Реалізація паттерну Singleton через перевизначення `__new__`.
    - Роль параметра `self`. Чому Python вимагає його вказувати явно. Що відбувається під час виклику `obj.method()` (зв'язані та незв'язані методи).
    - Динамічна природа об'єктів: робота з локальним простором імен `__dict__`.
    - Оптимізація пам'яті через `__slots__`: як відключити `__dict__`, чому це прискорює доступ до атрибутів та економить RAM (з конкретними вимірами пам'яті через `sys.getsizeof` та `tracemalloc`).
- **Візуалізація:** Схема зв'язків екземпляра об'єкта, його класу та метакласу `type` в пам'яті.

### Модуль 2: Структурні взаємозв'язки та Поліморфізм

#### Стаття 3: Спадкування, MRO та суперсила super()

- **Файл:** `03.inheritance-mro.md`
- **Орієнтовний обсяг:** 3 500–4 500 слів
- **Основні теми:**
    - Одиночне спадкування. Перевизначення методів (Method Overriding).
    - Робота з `super()`: чому це не просто «виклик батьківського класу», а проксі-об'єкт для обходу MRO. Передача аргументів у `super().__init__`.
    - Множинне спадкування та проблема «діаманта» (Diamond Problem).
    - Глибокий розбір алгоритму **MRO (Method Resolution Order)** та лінеаризації **C3**. Покроковий ручний розрахунок черги пошуку методів для складної ієрархії.
    - Використання `__mro__` класу для налагодження конфліктів спадкування.
    - Патерн **Mixins (домішки)**: проектування гнучких класів без створення глибоких ієрархій (наприклад, `JSONSerializableMixin`, `LoggingMixin`).
- **Візуалізація:** Інтерактивна схема MRO з обходом графів.

#### Стаття 4: Абстракція: ABC проти Статичних Протоколів (PEP 544)

- **Файл:** `04.abstraction-protocols.md`
- **Орієнтовний обсяг:** 3 000–3 800 слів
- **Основні теми:**
    - Номінативна типізація (Nominal Subtyping) проти структурної типізації (Structural Subtyping / Duck Typing).
    - Модуль `abc` (Abstract Base Classes). Створення абстрактних класів через `ABC` та декоратор `@abstractmethod`.
    - Заборона інстанціювання абстрактних класів. Абстрактні властивості (`@property` + `@abstractmethod`), абстрактні classmethods та staticmethods.
    - Сучасна альтернатива: **Protocols (PEP 544)** з модуля `typing`. Опис інтерфейсів без явного наслідування.
    - Статична перевірка типів за допомогою `mypy` при роботі з Protocols та ABC.
- **Порівняльна таблиця:** Детальний аналіз `ABC` vs `Protocol` (раннє зв'язування, час виконання, статичний аналіз, гнучкість).

### Модуль 3: Механізми поведінки та Метапрограмування

#### Стаття 5: Магічні методи (Dunder) та Емуляція протоколів

- **Файл:** `05.dunder-methods.md`
- **Орієнтовний обсяг:** 4 000–5 000 слів
- **Основні теми:**
    - Що таке Dunder (Double Underscore) методи та як вони реалізують протоколи мови Python.
    - **Протокол строкового представлення:** `__str__` (для користувачів) проти `__repr__` (детально для розробників/налагодження). Правило: `eval(repr(obj)) == obj`.
    - **Протокол порівняння та хешування:** `__eq__`, `__lt__`, `__le__` тощо. Чому при перевизначенні `__eq__` потрібно перевизначати `__hash__`, якщо об'єкт має бути hashable (наприклад, для використання як ключ у dict або в set).
    - **Арифметичний протокол:** `__add__`, `__radd__`, `__iadd__`. Що таке "right-hand" та "in-place" оператори.
    - **Емуляція контейнерів (Container Protocol):** `__len__`, `__getitem__`, `__setitem__`, `__delitem__`, `__contains__`. Створення власних custom колекцій.
    - **Протокол виклику:** `__call__` для створення callable об'єктів (об'єктів-функцій).
    - **Протокол контекстного менеджера:** `__enter__` та `__exit__`. Обробка виключень всередині `__exit__`.
- **Компоненти Docus:** `::field-group` з детальним описом сигнатур кожного сімейства dunder-методів.

#### Стаття 6: Декоратори та Керування життєвим циклом методів

- **Файл:** `06.decorators-static-class.md`
- **Орієнтовний обсяг:** 2 800–3 500 слів
- **Основні теми:**
    - Декоратори `@staticmethod` та `@classmethod`: концептуальна різниця, коли що використовувати. Чому `@classmethod` ідеальний для альтернативних конструкторів (`from_dict`, `from_csv`).
    - Як працюють декоратори методів: збереження стану, передача `self`.
    - Декорування цілих класів (Class Decorators): модифікація атрибутів класу при імпорті/ініціалізації.
    - Класи як декоратори (Callable instances): реалізація декораторів зі збереженням стану у вигляді екземплярів класів (наприклад, рейтінг-лімітер або кеш).
- **Практичний приклад:** Створення декоратора авторизації доступу до методів класу на основі ролей користувача.

#### Стаття 7: Дескриптори: Магія доступу до атрибутів

- **Файл:** `07.descriptors.md`
- **Орієнтовний обсяг:** 3 500–4 500 слів
- **Основні теми:**
    - Протокол дескрипторів: методи `__get__`, `__set__` та `__delete__`. Метод `__set_name__` (введений в Python 3.6 для отримання імені змінної без метакласів).
    - Порядок пошуку атрибутів у Python: `__getattribute__`, дескриптори даних (Data Descriptors), `__dict__` об'єкта, дескриптори без даних (Non-data Descriptors), `__getattr__`.
    - Чому `@property`, `@classmethod` та звичайні функції під капотом є дескрипторами.
    - Сценарії використання:
        - Валідація типів та діапазонів значень для полів класу (наприклад, `IntegerRangeDescriptor`).
        - Ліниве обчислення атрибутів (Lazy / Cached Property).
        - Емуляція ORM-полів (декларативне визначення колонок бази даних).
- **Візуалізація:** Алгоритм вирішення імен атрибутів (Lookup Chain).

#### Стаття 8: Метакласи та Динамічне створення класів

- **Файл:** `08.metaclasses.md`
- **Орієнтовний обсяг:** 4 000–5 000 слів
- **Основні теми:**
    - Розуміння того, що класи — це об'єкти. Динамічне створення класів «на льоту» за допомогою конструктора `type(name, bases, dict)`.
    - Що таке метаклас. Роль `type` як базового метакласу за замовчуванням.
    - Написання власного метакласу: наслідування від `type`, перевизначення `__new__` та `__init__` метакласу. Контроль та модифікація процесу створення класів (наприклад, автоматичне додавання префіксів, валідація наявності методів).
    - Практичне використання метакласів: реєстрація плагінів (Plugin Registry), автоматичне логування створення класів, побудова спрощеної декларативної ORM системи.
    - Сучасна альтернатива метакласам: метод `__init_subclass__` (PEP 487). Чому в 90% випадків замість складних метакласів краще використовувати `__init_subclass__`.
- **Компоненти Docus:** `::debugger-view` для відстеження черги виконання методів `__new__` метакласу, `__new__` класу, `__init__` метакласу та `__init__` класу.

### Модуль 4: Сучасні стандарти та Архітектура

#### Стаття 9: Dataclasses, NamedTuple та сучасні контейнери

- **Файл:** `09.modern-containers.md`
- **Орієнтовний обсяг:** 3 000–3 800 слів
- **Основні теми:**
    - Чому звичайні класи для збереження даних — це оверхед написання бойлерплейту (`__init__`, `__repr__`, `__eq__`).
    - Розбір модуля `dataclasses` (PEP 557): декоратор `@dataclass`, генерація коду під капотом.
    - Глибокі налаштування `@dataclass`: `frozen=True` (незмінність), `slots=True` (оптимізація пам'яті), `order=True` (порівняння).
    - Робота з функцією `field()`: налаштування значень за замовчуванням через `default_factory`, виключення полів з представлення чи порівняння.
    - Метод `__post_init__` для додаткової валідації та обчислення залежних полів.
    - Альтернативи: `typing.NamedTuple` (імутабельність на рівні tuple, швидкість) та порівняння з `collections.namedtuple`.
    - Використання `enum.Enum` та `enum.IntEnum`, `enum.auto` для створення перелічуваних типів.
- **Порівняльна таблиця:** Звичайний клас vs Dataclass vs NamedTuple vs TypedDict (пам'ять, швидкість створення об'єкта, мутабельність, синтаксис).

#### Стаття 10: SOLID та Паттерни проектування в динамічному Python

- **Файл:** `10.solid-design-patterns.md`
- **Орієнтовний обсяг:** 4 000–5 000 слів
- **Основні теми:**
    - Як SOLID принципи адаптуються під динамічну типізацію Python (наприклад, Liskov Substitution Principle та Duck Typing).
    - **Dependency Injection (впровадження залежностей)** в Python без важких ентерпрайз-фреймворків. Чистий код за допомогою type hinting.
    - Реалізація класичних GoF (Gang of Four) патернів з урахуванням специфіки Python:
        - **Singleton:** Чому класичний Java/C# підхід є антипатерном в Python. Реалізація через модуль (модулі в Python є синглтонами за замовчуванням), декоратор або метаклас.
        - **Factory / Abstract Factory:** Використання першокласних функцій/класів (First-class citizens) для спрощення фабрик.
        - **Strategy:** Реалізація паттерну без створення додаткових класів інтерфейсів — передача функцій або callable об'єктів як стратегій.
        - **Observer:** Реалізація подієвої моделі на основі слабких посилань (`weakref`) для запобігання витокам пам'яті.
- **Візуалізація:** UML діаграми реалізованих патернів разом із відповідними Python-схемами.

### Модуль 5: Конкурентність та паралелізм у Python

> Один із найважливіших практичних розділів. Охоплює три принципово різні моделі: кооперативну конкурентність (asyncio), потокову конкурентність (threading) та справжній паралелізм (multiprocessing). Матеріал побудований «від фундаменту» — з окремою статтею про GIL.

#### Стаття 11: GIL та модель конкурентності CPython

- **Файл:** `11.gil-concurrency-intro.md`
- **Орієнтовний обсяг:** 3 000–4 000 слів
- **Основні теми:**
    - Concurrency vs Parallelism: концептуальна різниця. Чому «одночасно» — не завжди «паралельно».
    - I/O-bound vs CPU-bound задачі: як характер задачі визначає вибір інструменту.
    - Що таке GIL (Global Interpreter Lock): навіщо він потрібен, як захищає підрахунок посилань CPython.
    - Як GIL реалізований: `check_interval`, `sys.getswitchinterval()`, поведінка при I/O.
    - Чому GIL не є проблемою для I/O-bound коду і є проблемою для CPU-bound.
    - Benchmarks: порівняння часу виконання I/O та CPU задач у різних моделях конкурентності.
    - Огляд трьох підходів і шпаргалка вибору: `threading` / `multiprocessing` / `asyncio`.
- **Візуалізація:** Порівняльна діаграма виконання потоків під GIL (time-slice схема) та без нього.

#### Стаття 12: Потоки та `threading` — конкурентність для I/O-bound задач

- **Файл:** `12.threading.md`
- **Орієнтовний обсяг:** 4 000–5 000 слів
- **Основні теми:**
    - `threading.Thread`: створення, запуск (`start()`), очікування (`join()`), `daemon`-потоки.
    - Стан гонки (Race Condition): що це, як виникає, як відтворити навмисно.
    - Примітиви синхронізації: `Lock`, `RLock` (реєнтерабельне блокування), `Semaphore`, `Event`, `Condition`, `Barrier`.
    - Потокобезпечні структури даних: `queue.Queue`, `queue.LifoQueue`.
    - `ThreadPoolExecutor` (з `concurrent.futures`): пул потоків, `submit()`, `map()`, `Future`.
    - Реальний патерн: завантаження кількох URL паралельно через `ThreadPoolExecutor`.
    - `threading.local()`: thread-local storage для ізоляції стану між потоками.
    - Проблема deadlock: як виникає і як уникати.
- **Компоненти Docus:** `::debugger-view` для відображення стану блокування потоків.

#### Стаття 13: Процеси та `multiprocessing` — справжній паралелізм

- **Файл:** `13.multiprocessing.md`
- **Орієнтовний обсяг:** 4 000–5 000 слів
- **Основні теми:**
    - `multiprocessing.Process`: відмінність від потоків (окремий GIL, окрема пам'ять).
    - Методи запуску: `spawn`, `fork`, `forkserver` — відмінності і платформна сумісність.
    - `ProcessPoolExecutor` і `Pool`: `map()`, `starmap()`, `apply_async()` для CPU-bound задач.
    - Міжпроцесна комунікація (IPC): `Queue`, `Pipe`, `Value`, `Array`, `Manager`.
    - `shared_memory` (Python 3.8+): спільна пам'ять без серіалізації.
    - Проблема pickling: що можна серіалізувати між процесами, а що — ні.
    - Практичний приклад: паралельна обробка зображень / числодробарка через `Pool`.
    - Порівняння `Pool.map` vs `ProcessPoolExecutor` — коли що вибирати.
- **Компоненти Docus:** `::terminal-preview` з реальними замірами speedup на N ядрах.

#### Стаття 14: `asyncio` — кооперативна конкурентність та event loop

- **Файл:** `14.asyncio.md`
- **Орієнтовний обсяг:** 5 000–6 000 слів
- **Основні теми:**
    - Модель event loop: як один потік обслуговує тисячі з'єднань через callbacks і coroutines.
    - `async def` та `await`: синтаксис, семантика, чим coroutine відрізняється від генератора.
    - `asyncio.Task` і `asyncio.Future`: різниця, `asyncio.create_task()`, `asyncio.gather()`, `asyncio.wait()`.
    - Синхронізація в asyncio: `asyncio.Lock`, `asyncio.Semaphore`, `asyncio.Event`, `asyncio.Queue`.
    - `asyncio.timeout()` і `asyncio.wait_for()`: обмеження часу виконання.
    - Інтеграція з синхронним кодом: `loop.run_in_executor()` для виклику блокуючих функцій.
    - Практичний приклад: асинхронний HTTP-клієнт (`aiohttp`/`httpx`) для паралельних запитів.
    - Типові антипатерни: блокуючі виклики всередині coroutine, забуті `await`, неправильне використання `asyncio.run()`.
- **Компоненти Docus:** `::debugger-view` для візуалізації черги event loop.

## План верифікації та написання матеріалів

1. **Створення шаблонів статей:** Перед написанням першої статті підготувати базову структуру markdown з YAML frontmatter.
2. **Валідація PlantUML:** Кожна діаграма повинна рендеритися коректно, не містити HTML тегів у назвах зв'язків та використовувати сучасну колірну палітру (Tailwind Indigo/Emerald/Slate).
3. **Запуск тестових скриптів:** Весь код, що наводиться в статтях, має бути протестованим на версіях Python 3.10+. Скрипти для вимірювання продуктивності/пам'яті (наприклад, у статтях про `__slots__` та дескриптори) повинні бути збережені в робочій директорії для можливості їх швидкого перезапуску.
4. **Контроль якості:** Перевіряти відповідність стилю Docus компонентів (коректне закриття тегів `::` та унікальність ID елементів).
