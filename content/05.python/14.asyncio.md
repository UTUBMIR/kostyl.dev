# asyncio — кооперативна конкурентність та event loop

## Проблема: 10 000 з'єднань, один сервер

Уявіть, що ви будуєте чат-сервер. До нього одночасно підключені 10 000 користувачів. Кожен час від часу надсилає повідомлення та очікує відповідь. Що відбудеться, якщо обробляти кожне з'єднання в окремому потоці?

```
10 000 потоків × (512 КБ стеку потоку) ≈ 5 ГБ пам'яті лише на стеки
+ накладні витрати ОС на переключення контексту між 10 000 потоків
+ Linux за замовчуванням обмежує кількість потоків до ~32 000
```

Система почне захлинатись задовго до того, як досягне реальних 10 000 з'єднань. Nginx вирішив цю проблему ще в 2002 році — не потоками, а **подієвим циклом** (event loop). Python отримав офіційний event loop у вигляді `asyncio` (PEP 3156, Python 3.4, суттєво оновлений у 3.7+).

::card-group

::card{title="Потоки: preemptive" icon="i-heroicons-cpu-chip"}
ОС перемикає потоки примусово через таймер. Кожен потік споживає ~512 КБ пам'яті на стек. Тисячі потоків — мільярди байт і мільйони переключень контексту на секунду.
::

::card{title="asyncio: cooperative" icon="i-heroicons-arrow-path"}
Корутини добровільно передають управління через `await`. Один потік — один стек. Тисячі корутин споживають мінімум пам'яті. Переключення відбувається лише в точках `await`.
::

::card{title="Де asyncio виграє" icon="i-heroicons-bolt"}
Веб-сервери, чат-системи, API-клієнти, парсери, WebSocket-сервери — все, де потрібно обробляти тисячі одночасних I/O-операцій з мінімальними ресурсами.
::

::card{title="Де asyncio не допоможе" icon="i-heroicons-x-circle"}
CPU-bound обчислення — event loop заблокується так само, як і будь-який інший потік. Для цього — `multiprocessing` або `run_in_executor`.
::

::

## Частина I: Модель event loop — як це працює зсередини

### Від callbacks до coroutines: еволюція асинхронного Python

Перш ніж `async/await`, у Python (і до сьогодні в JavaScript) асинхронність реалізовувалась через **callbacks** — функції, що передаються як аргументи і викликаються коли операція завершиться:

```python
# callback_style.py — старий підхід (asyncio низькорівневий API)
import asyncio


def on_data_received(data: bytes) -> None:
    print(f"Отримано: {data.decode()}")


def on_connection_made(transport) -> None:
    transport.write(b"Hello!\n")


# Callback-based API існує в asyncio, але напряму майже не використовується
# Натомість є coroutines — набагато чистіший синтаксис
```

Callback-підхід призводить до «callback hell» — глибоко вкладеного, важкочитабельного коду. Python 3.5 ввів `async/await` — синтаксичний цукор над генераторами, що дозволяє писати асинхронний код у **лінійному стилі**, ніби він синхронний.

### Анатомія event loop

Event loop — це нескінченний цикл, що виконує три речі:

```
while True:
    1. Опитати ОС: які I/O-операції завершились? (epoll/kqueue/IOCP)
    2. Викликати callbacks для завершених операцій
    3. Виконати наступний крок готових корутин
```

```python
# event_loop_anatomy.py
import asyncio


async def say_hello() -> None:
    print("Привіт!")           # крок 1
    await asyncio.sleep(1)     # ← Тут корутина ДОБРОВІЛЬНО повертає управління
    print("Пройшла секунда")   # крок 2 — після того, як sleep завершився


async def count() -> None:
    for i in range(3):
        print(f"  Рахую: {i}")
        await asyncio.sleep(0.3)  # теж звільняє управління


async def main() -> None:
    # Запускаємо обидві корутини КОНКУРЕНТНО
    await asyncio.gather(say_hello(), count())


# asyncio.run() створює event loop, запускає main() і закриває loop
asyncio.run(main())
```

### Що відбувається у точці `await`

```python
# await_mechanics.py
import asyncio
import time


async def fast_io() -> str:
    """Симулює швидку I/O операцію (10 мс)."""
    await asyncio.sleep(0.01)
    return "fast result"


async def slow_io() -> str:
    """Симулює повільну I/O операцію (500 мс)."""
    await asyncio.sleep(0.5)
    return "slow result"


async def demonstrate_await() -> None:
    print("=== Послідовно (await кожного окремо) ===")
    t0 = time.perf_counter()
    r1 = await fast_io()   # Чекаємо 10мс, потім продовжуємо
    r2 = await slow_io()   # Чекаємо 500мс, потім продовжуємо
    print(f"Результати: {r1}, {r2}")
    print(f"Час: {time.perf_counter() - t0:.2f}s")  # ~0.51s

    print("\n=== Конкурентно (gather) ===")
    t0 = time.perf_counter()
    r1, r2 = await asyncio.gather(fast_io(), slow_io())
    print(f"Результати: {r1}, {r2}")
    print(f"Час: {time.perf_counter() - t0:.2f}s")  # ~0.50s — лише slow_io визначає!


asyncio.run(demonstrate_await())
```

## Частина II: `async def` і `await` — синтаксис і семантика

### Що таке корутина під капотом

`async def` оголошує **корутинну функцію**. Виклик такої функції **не виконує** її тіло — він повертає об'єкт корутини:

```python
# coroutine_anatomy.py
import asyncio
import inspect


async def greet(name: str) -> str:
    await asyncio.sleep(0.1)
    return f"Привіт, {name}!"


# Виклик coroutine function → coroutine object (не виконується!)
coro = greet("Іван")
print(type(coro))           # <class 'coroutine'>
print(inspect.iscoroutine(coro))  # True
print(inspect.isawaitable(coro))  # True

# Сама по собі корутина нічого не робить:
# coro  ← тільки об'єкт, жодного виводу

# Щоб виконати — потрібен event loop через await або asyncio.run():
result = asyncio.run(greet("Марія"))
print(result)  # "Привіт, Марія!"

# Або зсередини іншої async-функції:
async def main():
    result = await greet("Олег")  # тепер виконується
    print(result)

asyncio.run(main())
```

### Coroutine vs Generator — схожість і відмінність

Синтаксично корутини `async def / await` схожі на генератори `def / yield`. Це не збіг — під капотом CPython корутини реалізовані поверх механізму генераторів (PEP 492). Але семантика принципово різна:

| Характеристика | Generator (`yield`) | Coroutine (`await`) |
|---|---|---|
| Оголошення | `def f(): yield` | `async def f(): await` |
| Повертає | Значення через `yield` | Результат через `return` |
| Призначення | Ліниве генерування послідовностей | Асинхронні I/O-операції |
| Виконання | `next()` / `for` цикл | `await` / event loop |
| Можна `yield` | Так | Ні (тільки `await`) |
| Можна `await` | Ні | Так |

```python
# coro_vs_gen.py

# Генератор — виробник значень
def fibonacci():
    a, b = 0, 1
    while True:
        yield a          # призупиняє і повертає значення
        a, b = b, a + b


# Корутина — асинхронна операція
async def fetch_page(url: str) -> bytes:
    import asyncio
    await asyncio.sleep(0.1)   # призупиняє і чекає завершення IO
    return b"<html>...</html>" # повертає результат при return, а не yield


# Асинхронний генератор — поєднання обох!
async def async_range(n: int):
    for i in range(n):
        await asyncio.sleep(0.01)  # асинхронна пауза між значеннями
        yield i                     # генерує значення


async def consume_async_gen():
    async for value in async_range(5):  # async for!
        print(value)
```

### Що можна `await`

`await` можна використовувати з будь-яким **awaitable** об'єктом. У Python є три категорії:

::field-group

::field{name="Coroutine" type="async def функція"}
Найпоширеніший тип. `await some_coroutine()` — виконати корутину і отримати її результат. Виклик без `await` просто створює об'єкт, але не виконує його.
::

::field{name="Task" type="asyncio.Task"}
Обгортка навколо корутини, що **вже запущена** event loop'ом. Створюється через `asyncio.create_task()`. На відміну від корутини, Task починає виконуватись одразу після створення, не чекаючи `await`.
::

::field{name="Future" type="asyncio.Future"}
Низькорівневий об'єкт-обіцянка результату. Зазвичай не створюється вручну — повертається деякими низькорівневими API. `Task` є підкласом `Future`.
::

::

```python
# awaitables_demo.py
import asyncio


async def compute() -> int:
    await asyncio.sleep(0.1)
    return 42


async def main():
    # 1. Await coroutine (послідовно)
    result = await compute()
    print(f"Coroutine: {result}")

    # 2. Await Task (конкурентно — task запускається ОДРАЗУ при create_task)
    task = asyncio.create_task(compute())  # ← вже в черзі виконання!
    print("Task створено, інший код може виконуватись тут...")
    result = await task  # чекаємо завершення
    print(f"Task: {result}")

    # 3. Future — встановлюємо результат вручну
    loop = asyncio.get_event_loop()
    future: asyncio.Future[str] = loop.create_future()
    loop.call_soon(future.set_result, "future value")  # встановимо результат незабаром
    result = await future
    print(f"Future: {result}")


asyncio.run(main())
```

### `asyncio.run()` — правильна точка входу

```python
# asyncio_run_usage.py
import asyncio


async def main() -> int:
    print("Старт")
    await asyncio.sleep(0.1)
    print("Фініш")
    return 0


# ✅ Сучасний підхід (Python 3.7+)
exit_code = asyncio.run(main())

# ❌ Застарілий підхід (не використовуйте в новому коді):
# loop = asyncio.get_event_loop()
# loop.run_until_complete(main())
# loop.close()
```

`asyncio.run()` під капотом:
1. Створює новий event loop
2. Запускає передану корутину до завершення
3. Закриває event loop і звільняє ресурси
4. Повертає результат корутини

::warning
`asyncio.run()` не можна викликати зсередині вже запущеного event loop — наприклад, у Jupyter Notebook. У Jupyter вже є запущений loop, тому треба використовувати `await main()` напряму або бібліотеку `nest_asyncio`.
::

## Частина III: `Task` і `Future` — конкурентне виконання

### `asyncio.create_task()` — запустити корутину «у фоні»

`create_task()` — ключова функція для справжньої конкурентності в asyncio. Вона загортає корутину в `Task` і **одразу планує її виконання** у event loop, не чекаючи `await`:

```python
# create_task_demo.py
import asyncio
import time


async def download(name: str, delay: float) -> str:
    print(f"[{name}] Починаю завантаження...")
    await asyncio.sleep(delay)
    print(f"[{name}] Готово! ({delay}s)")
    return f"{name}: {delay}s"


async def sequential() -> None:
    """Послідовне виконання — довго."""
    t0 = time.perf_counter()
    r1 = await download("FileA", 1.0)
    r2 = await download("FileB", 1.5)
    r3 = await download("FileC", 0.8)
    print(f"Sequential: {time.perf_counter() - t0:.2f}s")  # ~3.3s


async def concurrent_with_tasks() -> None:
    """Конкурентне виконання через create_task."""
    t0 = time.perf_counter()

    # create_task() планує виконання ОДРАЗУ — всі три стартують "паралельно"
    task_a = asyncio.create_task(download("FileA", 1.0), name="task-A")
    task_b = asyncio.create_task(download("FileB", 1.5), name="task-B")
    task_c = asyncio.create_task(download("FileC", 0.8), name="task-C")

    # Чекаємо всі три — вони вже виконуються поки ми тут
    r1 = await task_a
    r2 = await task_b
    r3 = await task_c
    print(f"Concurrent: {time.perf_counter() - t0:.2f}s")  # ~1.5s (найповільніший)


asyncio.run(concurrent_with_tasks())
```

### `asyncio.gather()` — запустити і зібрати результати

`gather()` — зручна обгортка для запуску кількох корутин конкурентно і очікування **всіх** результатів:

```python
# gather_demo.py
import asyncio


async def fetch(url: str) -> tuple[str, int]:
    await asyncio.sleep(0.3)   # симулюємо мережевий запит
    return url, 200


async def main():
    urls = [
        "https://api.example.com/users",
        "https://api.example.com/posts",
        "https://api.example.com/comments",
        "https://api.example.com/tags",
    ]

    # ── gather з розпакуванням ────────────────────────────────────────────────
    (url1, s1), (url2, s2), (url3, s3), (url4, s4) = await asyncio.gather(
        *[fetch(u) for u in urls]
    )
    print(f"Отримано {s1} від {url1}")

    # ── gather з обробкою помилок (return_exceptions=True) ───────────────────
    async def risky_fetch(url: str) -> str:
        if "bad" in url:
            raise ValueError(f"Поганий URL: {url}")
        await asyncio.sleep(0.1)
        return f"OK: {url}"

    mixed_urls = ["good/1", "bad/url", "good/2"]
    results = await asyncio.gather(
        *[risky_fetch(u) for u in mixed_urls],
        return_exceptions=True,   # ← не кидає виключення, повертає його як результат
    )
    for url, result in zip(mixed_urls, results):
        if isinstance(result, Exception):
            print(f"  ✗ {url}: {result}")
        else:
            print(f"  ✓ {url}: {result}")


asyncio.run(main())
```

::important
За замовчуванням `asyncio.gather()` при помилці **скасовує** всі інші корутини і кидає виключення. `return_exceptions=True` змінює поведінку: помилки повертаються як звичайні результати у відповідній позиції списку, решта корутин продовжує виконуватись.
::

### `asyncio.wait()` — більше контролю над очікуванням

`asyncio.wait()` на відміну від `gather()` повертає дві множини: `done` (завершені) і `pending` (ще виконуються). Це дозволяє обробляти результати поступово або зупинитись після першого результату:

```python
# asyncio_wait_demo.py
import asyncio


async def task(name: str, delay: float) -> str:
    await asyncio.sleep(delay)
    return f"{name} ({delay}s)"


async def main():
    tasks = [
        asyncio.create_task(task("Alpha",   0.5)),
        asyncio.create_task(task("Beta",    1.2)),
        asyncio.create_task(task("Gamma",   0.8)),
        asyncio.create_task(task("Delta",   2.0)),
    ]

    # ── FIRST_COMPLETED: зупинитись після першого результату ─────────────────
    print("=== FIRST_COMPLETED ===")
    done, pending = await asyncio.wait(tasks, return_when=asyncio.FIRST_COMPLETED)
    for t in done:
        print(f"  Перший завершений: {t.result()}")
    print(f"  Ще виконуються: {len(pending)} задач")

    # Скасовуємо решту (необов'язково, але гарна практика)
    for t in pending:
        t.cancel()

    # ── ALL_COMPLETED: чекати всіх (поведінка як у gather) ───────────────────
    tasks2 = [
        asyncio.create_task(task("X", 0.3)),
        asyncio.create_task(task("Y", 0.6)),
    ]
    done, pending = await asyncio.wait(tasks2, return_when=asyncio.ALL_COMPLETED)
    for t in done:
        print(f"  Результат: {t.result()}")

    # ── timeout: чекати не більше N секунд ───────────────────────────────────
    tasks3 = [asyncio.create_task(task("Slow", 5.0))]
    done, pending = await asyncio.wait(tasks3, timeout=0.5)
    if pending:
        print(f"  Таймаут: {len(pending)} задач не завершились")
        for t in pending: t.cancel()


asyncio.run(main())
```

### `asyncio.as_completed()` — обробляти результати в міру готовності

```python
# as_completed_demo.py
import asyncio
import random


async def fetch_random(item_id: int) -> tuple[int, float]:
    delay = random.uniform(0.1, 1.0)
    await asyncio.sleep(delay)
    return item_id, delay


async def main():
    tasks = [fetch_random(i) for i in range(10)]

    # as_completed() — ітератор, що віддає futures в порядку ЗАВЕРШЕННЯ
    print("Результати в порядку завершення:")
    async for coro in asyncio.as_completed(tasks):
        item_id, delay = await coro
        print(f"  #{item_id:02d} завершено за {delay:.2f}s")


asyncio.run(main())
```

### `TaskGroup` — сучасний API (Python 3.11+)

Python 3.11 ввів `asyncio.TaskGroup` — структурований спосіб керувати групою задач зі строгою семантикою скасування:

```python
# task_group_demo.py
import asyncio


async def worker(name: str, delay: float) -> str:
    await asyncio.sleep(delay)
    print(f"  [Worker {name}] done")
    return name


async def main():
    # TaskGroup: якщо одна задача кидає виключення — всі інші скасовуються
    try:
        async with asyncio.TaskGroup() as tg:
            t1 = tg.create_task(worker("A", 0.3))
            t2 = tg.create_task(worker("B", 0.5))
            t3 = tg.create_task(worker("C", 0.8))
        # Тут всі задачі гарантовано завершені (або скасовані через помилку)
        print(f"Результати: {t1.result()}, {t2.result()}, {t3.result()}")
    except* ValueError as eg:
        # except* для ExceptionGroup (Python 3.11+)
        print(f"Помилки: {eg.exceptions}")


asyncio.run(main())
```

::tip
`TaskGroup` — це рекомендований спосіб роботи з групами задач у Python 3.11+. Він забезпечує строгу структуровану конкурентність: якщо будь-яка задача у групі завершується з помилкою, всі інші автоматично скасовуються. Це запобігає «загубленим» задачам, що продовжують виконуватись після помилки.
::

## Частина IV: Синхронізація в asyncio

Хоча asyncio використовує один потік, спільний стан між корутинами все одно може викликати стан гонки — якщо кілька корутин читають і модифікують один об'єкт між точками `await`. Для цього є асинхронні примітиви синхронізації.

### `asyncio.Lock` — взаємне виключення

```python
# async_lock_demo.py
import asyncio


# Спільний ресурс — лічильник
counter = 0
lock = asyncio.Lock()


async def increment(name: str, n: int) -> None:
    global counter
    for _ in range(n):
        # БЕЗ lock: між await sleep(0) і counter += 1 може вклинитись інша корутина
        async with lock:          # ← захоплюємо замок
            current = counter
            await asyncio.sleep(0)  # Симулюємо мінімальне перемикання
            counter = current + 1
    print(f"[{name}] завершено, counter={counter}")


async def main():
    global counter
    counter = 0

    await asyncio.gather(
        increment("A", 1000),
        increment("B", 1000),
    )
    print(f"Фінальний counter: {counter}")  # Завжди 2000 з lock


asyncio.run(main())
```

### `asyncio.Semaphore` — обмеження конкурентних операцій

Найтиповіший сценарій: обмежити кількість одночасних HTTP-запитів, щоб не перевантажити зовнішній API:

```python
# async_semaphore_demo.py
import asyncio
import time


MAX_CONCURRENT_REQUESTS = 3
semaphore = asyncio.Semaphore(MAX_CONCURRENT_REQUESTS)
active_count = 0


async def fetch_with_limit(url: str, session_id: int) -> tuple[str, int]:
    global active_count
    async with semaphore:        # максимум 3 одночасно
        active_count += 1
        print(f"  → [{session_id}] Запит: {url[-20:]}  (активних: {active_count})")
        try:
            await asyncio.sleep(0.5)  # симулюємо запит
            return url, 200
        finally:
            active_count -= 1
            print(f"  ← [{session_id}] Готово  (активних: {active_count})")


async def main():
    urls = [f"https://api.example.com/resource/{i}" for i in range(10)]

    t0 = time.perf_counter()
    results = await asyncio.gather(
        *[fetch_with_limit(url, i) for i, url in enumerate(urls)]
    )
    elapsed = time.perf_counter() - t0

    print(f"\nОброблено {len(results)} URL за {elapsed:.2f}s")
    print(f"(Без обмеження: ~0.5s; з Semaphore(3): ~{0.5 * (len(urls) // MAX_CONCURRENT_REQUESTS):.1f}s)")


asyncio.run(main())
```

### `asyncio.Event` — сигналізація між корутинами

```python
# async_event_demo.py
import asyncio


async def producer(event: asyncio.Event, data: list) -> None:
    """Підготовує дані і сигналізує про готовність."""
    print("[Producer] Готую дані...")
    await asyncio.sleep(1.0)
    data.extend([1, 2, 3, 4, 5])
    print("[Producer] Дані готові!")
    event.set()   # Сигналізуємо всім Consumer-ам


async def consumer(event: asyncio.Event, data: list, consumer_id: int) -> None:
    """Чекає готовності даних."""
    print(f"[Consumer-{consumer_id}] Чекаю даних...")
    await event.wait()   # Блокується до event.set()
    print(f"[Consumer-{consumer_id}] Отримав {len(data)} елементів: {data}")


async def main():
    ready_event = asyncio.Event()
    shared_data: list[int] = []

    await asyncio.gather(
        producer(ready_event, shared_data),
        consumer(ready_event, shared_data, 1),
        consumer(ready_event, shared_data, 2),
        consumer(ready_event, shared_data, 3),
    )


asyncio.run(main())
```

### `asyncio.Queue` — потокобезпечна черга для корутин

`asyncio.Queue` — асинхронний аналог `queue.Queue`. Ідеально для producer-consumer патерну між корутинами:

```python
# async_queue_demo.py
import asyncio
import random


async def producer(queue: asyncio.Queue, n_items: int) -> None:
    """Генерує завдання і кладе в чергу."""
    for i in range(n_items):
        item = random.randint(1, 100)
        await queue.put(item)          # Блокується якщо черга повна
        print(f"  [Producer] → {item} (у черзі: {queue.qsize()})")
        await asyncio.sleep(0.1)

    # Кладемо sentinel для кожного consumer
    await queue.put(None)
    await queue.put(None)
    print("  [Producer] Всі завдання передані")


async def consumer(queue: asyncio.Queue, name: str) -> None:
    """Бере завдання з черги і обробляє."""
    total = 0
    while True:
        item = await queue.get()       # Блокується якщо черга порожня
        if item is None:
            print(f"  [{name}] Завершую, сума={total}")
            queue.task_done()
            break
        total += item
        print(f"  [{name}] Обробляю {item}")
        await asyncio.sleep(0.2)       # Consumer повільніший за Producer
        queue.task_done()


async def main():
    # maxsize обмежує розмір черги (backpressure)
    queue: asyncio.Queue[int | None] = asyncio.Queue(maxsize=5)

    await asyncio.gather(
        producer(queue, 8),
        consumer(queue, "Consumer-1"),
        consumer(queue, "Consumer-2"),
    )
    # queue.join() — чекає поки всі task_done() викличуть для кожного put()
    # await queue.join()


asyncio.run(main())
```

### Таймаути: `asyncio.timeout()` і `asyncio.wait_for()`

Контроль часу виконання — критично важливий у мережевому коді:

```python
# async_timeout_demo.py
import asyncio


async def slow_operation() -> str:
    """Операція, що може зайняти довго."""
    await asyncio.sleep(5.0)
    return "повільний результат"


async def main():
    # ── asyncio.timeout() (Python 3.11+) — context manager ──────────────────
    try:
        async with asyncio.timeout(1.0):  # максимум 1 секунда
            result = await slow_operation()
    except asyncio.TimeoutError:
        print("asyncio.timeout: операція перевищила 1s")

    # ── asyncio.wait_for() — обгортка для однієї корутини ───────────────────
    try:
        result = await asyncio.wait_for(slow_operation(), timeout=1.0)
    except asyncio.TimeoutError:
        print("wait_for: операція перевищила 1s")

    # ── Динамічне оновлення дедлайну ─────────────────────────────────────────
    async def fetch_with_retry(url: str, attempts: int = 3) -> str:
        for attempt in range(1, attempts + 1):
            try:
                async with asyncio.timeout(2.0 * attempt):  # збільшуємо таймаут
                    await asyncio.sleep(1.5)  # симулюємо запит
                    return f"OK from {url} (attempt {attempt})"
            except asyncio.TimeoutError:
                print(f"  Спроба {attempt} перевищила таймаут")
        raise RuntimeError(f"Всі {attempts} спроб провалились")

    result = await fetch_with_retry("https://api.example.com")
    print(f"Результат: {result}")


asyncio.run(main())
```

::tip
Завжди встановлюйте таймаути для мережевих операцій. Без таймауту корутина може чекати відповіді нескінченно — і event loop буде заблокований для цієї задачі (хоча інші задачі продовжуватимуть виконуватись). `asyncio.timeout()` є пріоритетним над `wait_for()` в Python 3.11+, оскільки підтримує оновлення дедлайну через `Timeout.reschedule()`.
::

## Частина V: Інтеграція з синхронним кодом

### `loop.run_in_executor()` — блокуючий код без блокування event loop

Головна небезпека asyncio — **будь-який блокуючий виклик зупиняє весь event loop**. `time.sleep()`, `requests.get()`, важкий CPU-код — все це заблокує всі інші корутини на час виконання. Рішення — `run_in_executor()`, що виконує блокуючу функцію у пулі потоків (або процесів), не блокуючи event loop:

```python
# run_in_executor_demo.py
import asyncio
import time
import concurrent.futures
import urllib.request


# ── Приклад 1: Синхронна бібліотека у потоці ──────────────────────────────────
def blocking_http_get(url: str) -> bytes:
    """Стара синхронна функція — блокує потік."""
    with urllib.request.urlopen(url, timeout=10) as r:
        return r.read()


def blocking_cpu_work(n: int) -> int:
    """CPU-важка операція — блокує потік."""
    time.sleep(0.1)  # Симулюємо CPU-роботу
    return sum(i * i for i in range(n))


async def main():
    loop = asyncio.get_event_loop()

    # ── ThreadPoolExecutor: для I/O-bound блокуючого коду ────────────────────
    with concurrent.futures.ThreadPoolExecutor(max_workers=5) as thread_pool:
        # run_in_executor запускає у потоці і повертає awaitable
        result = await loop.run_in_executor(
            thread_pool,
            blocking_http_get,
            "https://httpbin.org/get"
        )
        print(f"HTTP відповідь: {len(result)} байт")

        # Запускаємо кілька конкурентно
        urls = ["https://httpbin.org/delay/0.3"] * 5
        tasks = [
            loop.run_in_executor(thread_pool, blocking_http_get, url)
            for url in urls
        ]
        results = await asyncio.gather(*tasks)
        print(f"Отримано {len(results)} відповідей конкурентно")

    # ── ProcessPoolExecutor: для CPU-bound блокуючого коду ───────────────────
    with concurrent.futures.ProcessPoolExecutor(max_workers=4) as proc_pool:
        # Запускаємо CPU-важку задачу у процесі, не блокуємо event loop
        result = await loop.run_in_executor(
            proc_pool,
            blocking_cpu_work,
            1_000_000
        )
        print(f"CPU результат: {result:,}")

    # ── None executor: використовує default ThreadPoolExecutor ───────────────
    result = await loop.run_in_executor(None, blocking_cpu_work, 500_000)
    print(f"Default executor: {result:,}")


asyncio.run(main())
```

::tip
Правило вибору executor: якщо блокуючий код — I/O (стара бібліотека, `requests`), використовуйте `ThreadPoolExecutor`. Якщо блокуючий код — CPU-важка операція, використовуйте `ProcessPoolExecutor`. Для більшості сучасних бібліотек (aiohttp, asyncpg, aiobotocore) `run_in_executor` не потрібен — вони вже асинхронні.
::

```python
    # Python також покаже RuntimeWarning: coroutine 'fetch_data' was never awaited
```

### Антипатерн 3: Послідовне `await` замість `gather`

```python
import asyncio


async def fetch(n: int) -> int:
    await asyncio.sleep(1.0)
    return n


async def bad_concurrent():
    # ❌ НЕПРАВИЛЬНО: це НЕ конкурентно — виконується послідовно!
    # Загальний час: ~3 секунди
    r1 = await fetch(1)
    r2 = await fetch(2)
    r3 = await fetch(3)
    return [r1, r2, r3]


async def good_concurrent():
    # ✅ ПРАВИЛЬНО: gather запускає всі три конкурентно
    # Загальний час: ~1 секунда
    return await asyncio.gather(fetch(1), fetch(2), fetch(3))
```

### Антипатерн 4: `asyncio.run()` у вже запущеному loop

```python
import asyncio

    return "result"
```

## Частина VII: Реальний патерн — асинхронний веб-скрапер

Об'єднаємо все вивчене: асинхронний скрапер з обмеженням конкурентності, таймаутами, повторними спробами та статистикою:

```python
# async_scraper.py
from __future__ import annotations

import asyncio
import time
import urllib.request
from dataclasses import dataclass, field
from collections import defaultdict
import concurrent.futures


@dataclass
class ScrapeResult:
    url: str
    success: bool
    status_code: int = 0
    size_bytes: int = 0
    elapsed: float = 0.0
    attempts: int = 0
    error: str = ""


@dataclass
class ScraperStats:
    total: int = 0
    successful: int = 0
    failed: int = 0
    total_bytes: int = 0
    total_time: float = 0.0
    errors: dict[str, int] = field(default_factory=lambda: defaultdict(int))


def _sync_fetch(url: str) -> tuple[int, bytes]:
    """Синхронний fetch для run_in_executor."""
    with urllib.request.urlopen(url, timeout=10) as r:
        return r.status, r.read()


async def fetch_url(
    url: str,
    semaphore: asyncio.Semaphore,
    executor: concurrent.futures.ThreadPoolExecutor,
    max_retries: int = 3,
    retry_delay: float = 0.5,
) -> ScrapeResult:
    """Завантажує URL з обмеженням конкурентності, таймаутом і retry."""
    loop = asyncio.get_event_loop()

    for attempt in range(1, max_retries + 1):
        async with semaphore:    # Обмежуємо паралельні запити
            start = time.perf_counter()
            try:
                async with asyncio.timeout(8.0):
                    status, data = await loop.run_in_executor(
                        executor, _sync_fetch, url
                    )
                elapsed = time.perf_counter() - start
                return ScrapeResult(
                    url=url,
                    success=True,
                    status_code=status,
                    size_bytes=len(data),
                    elapsed=elapsed,
                    attempts=attempt,
                )
            except asyncio.TimeoutError:
                error_msg = f"Таймаут (спроба {attempt})"
            except Exception as e:
                error_msg = f"{type(e).__name__}: {e} (спроба {attempt})"

            if attempt < max_retries:
                await asyncio.sleep(retry_delay * attempt)  # Exponential backoff

    return ScrapeResult(
        url=url, success=False,
        elapsed=time.perf_counter() - start,
        attempts=max_retries, error=error_msg,
    )


async def scrape_all(
    urls: list[str],
    max_concurrent: int = 10,
    max_retries: int = 3,
    verbose: bool = True,
) -> tuple[list[ScrapeResult], ScraperStats]:
    """Асинхронно завантажує всі URL з обмеженням конкурентності."""
    semaphore = asyncio.Semaphore(max_concurrent)
    stats = ScraperStats(total=len(urls))

    with concurrent.futures.ThreadPoolExecutor(max_workers=max_concurrent) as executor:
        tasks = [
            fetch_url(url, semaphore, executor, max_retries)
            for url in urls
        ]

        results: list[ScrapeResult] = []
        t_start = time.perf_counter()

        async for coro in asyncio.as_completed(tasks):
            result = await coro
            results.append(result)

            if result.success:
                stats.successful += 1
                stats.total_bytes += result.size_bytes
                if verbose:
                    print(
                        f"  ✓ [{result.attempts} спр.] "
                        f"{result.url[-45:]:<45} "
                        f"{result.size_bytes:>7,}b  {result.elapsed:.2f}s"
                    )
            else:
                stats.failed += 1
                stats.errors[result.error] += 1
                if verbose:
                    print(f"  ✗ {result.url[-45:]}: {result.error}")

        stats.total_time = time.perf_counter() - t_start

    if verbose:
        print(f"\n{'─' * 60}")
        print(f"Всього URL:   {stats.total}")
        print(f"Успішно:      {stats.successful}")
        print(f"Помилок:      {stats.failed}")
        print(f"Отримано:     {stats.total_bytes:,} байт")
        print(f"Час:          {stats.total_time:.2f}s")
        if stats.total_time > 0:
            avg_seq = sum(r.elapsed for r in results if r.success)
            print(f"Прискорення:  {avg_seq / stats.total_time:.1f}x")

    return results, stats


if __name__ == "__main__":
    test_urls = [
        f"https://httpbin.org/delay/0.3?id={i}" for i in range(20)
    ]

    print(f"Скрапимо {len(test_urls)} URL (по ~0.3s кожен)\n")
    results, stats = asyncio.run(scrape_all(test_urls, max_concurrent=8))
```

## Підсумок: таблиця API та ключові принципи

### Повне API asyncio

| Функція/клас | Призначення |
|---|---|
| `asyncio.run(coro)` | Запустити корутину як точку входу |
| `asyncio.create_task(coro)` | Запустити корутину у фоні (Task) |
| `asyncio.gather(*coros)` | Конкурентно запустити кілька корутин, зібрати всі результати |
| `asyncio.wait(tasks, ...)` | Гнучке очікування задач з `return_when` |
| `asyncio.as_completed(coros)` | Ітератор результатів у порядку завершення |
| `asyncio.TaskGroup` | Структурована група задач (Python 3.11+) |
| `asyncio.sleep(n)` | Пауза без блокування event loop |
| `asyncio.timeout(n)` | Context manager для таймауту (Python 3.11+) |
| `asyncio.wait_for(coro, t)` | Обгортка з таймаутом |
| `asyncio.Lock` | Мютекс для async коду |
| `asyncio.Semaphore(n)` | Обмеження N одночасних операцій |
| `asyncio.Event` | Одноразовий сигнал між корутинами |
| `asyncio.Queue` | Потокобезпечна черга для корутин |
| `loop.run_in_executor()` | Запустити блокуючий код у ThreadPool/ProcessPool |
| `asyncio.current_task()` | Отримати поточний Task |
| `asyncio.all_tasks()` | Всі живі Task у поточному loop |

### Порівняння: `gather` vs `wait` vs `as_completed` vs `TaskGroup`

| Функція | Чекає всіх | Зупинка на помилці | Результати | Скасування |
|---|---|---|---|---|
| `gather()` | ✅ | За замовчуванням так | Список у порядку вводу | Вручну |
| `gather(return_exceptions=True)` | ✅ | Ні | Список із Exception | Вручну |
| `wait(ALL_COMPLETED)` | ✅ | Ні | `done`, `pending` множини | Вручну |
| `wait(FIRST_COMPLETED)` | ❌ | Ні | `done`, `pending` множини | Вручну |
| `as_completed()` | Поступово | Ні | Futures по одному | Вручну |
| `TaskGroup` | ✅ | ✅ Авто | Через `task.result()` | Авто при помилці |

### Ключові принципи asyncio

| Принцип | Деталь |
|---|---|
| `await` ≠ конкурентність | `await coro()` послідовно. Для конкурентності — `gather` або `create_task` |
| Немає `await` — немає виконання | Корутина без `await` — просто об'єкт у пам'яті |
| Блокуючий код = зупинка loop | `time.sleep`, `requests.get` блокують усі задачі. Використовуйте async-бібліотеки або `run_in_executor` |
| Завжди таймаути | `asyncio.timeout()` або `wait_for()` для кожного мережевого виклику |
| `TaskGroup` > `gather` | Python 3.11+: `TaskGroup` надає строгіші гарантії скасування |
| Семафор для rate limiting | `asyncio.Semaphore(N)` обмежує N паралельних запитів до API |
| `asyncio.run()` — один раз | Тільки на верхньому рівні програми, не всередині async функцій |
