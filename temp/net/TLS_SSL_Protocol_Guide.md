## Вступ до TLS/SSL

### Що таке TLS і SSL?

**SSL** (Secure Sockets Layer) — це криптографічний протокол, розроблений компанією Netscape у 1990-х роках для забезпечення безпечної передачі даних через Інтернет.

**TLS** (Transport Layer Security) — це наступник SSL, стандартизований протокол, який забезпечує конфіденційність, цілісність та автентифікацію даних.

**Простими словами:** TLS/SSL — це "конверт" для ваших даних. Коли ви відправляєте інформацію через Інтернет, TLS шифрує її так, щоб ніхто не міг прочитати або змінити по дорозі.

### Три головні цілі TLS/SSL:

#### 1. **Конфіденційність (Confidentiality)**

Дані зашифровані — ніхто не може їх прочитати по дорозі.

```
Без TLS:
Клієнт → "password123" → [ХАКЕР БАЧИТЬ!] → Сервер ❌

З TLS:
Клієнт → "jH8#mK@pL2$..." → [хакер бачить тільки шифр] → Сервер ✅
```

#### 2. **Цілісність (Integrity)**

Дані не можна змінити непомітно під час передачі.

```
Без TLS:
Клієнт → "переказати 100 грн" → [ХАКЕР ЗМІНЮЄ на "1000 грн"] → Сервер ❌

З TLS:
Клієнт → зашифровано + контрольна сума → [зміна виявлена!] → Сервер ✅
```

#### 3. **Автентифікація (Authentication)**

Впевненість, що ви спілкуєтесь з правильним сервером.

```
Без TLS:
Клієнт → [підключається до fake-bank.com] ❌

З TLS:
Клієнт → перевіряє сертифікат → [це справді bank.com!] → Сервер ✅
```

### Де використовується TLS/SSL?

-   🌐 **HTTPS** — захищений веб-трафік
-   📧 **Email** — SMTPS, IMAPS, POP3S
-   💬 **Messaging** — Signal, WhatsApp, Telegram
-   🔐 **VPN** — OpenVPN, деякі VPN протоколи
-   💾 **FTP** — FTPS (FTP over SSL)
-   🗄️ **Databases** — PostgreSQL, MySQL з SSL

### HTTPS = HTTP + TLS

```
HTTP (незахищений):
┌──────────┐                      ┌──────────┐
│ Браузер  │──── відкритий текст ───→│  Сервер  │
└──────────┘                      └──────────┘
        ↑
   [Хакер може читати/змінювати]

HTTPS (захищений):
┌──────────┐                      ┌──────────┐
│ Браузер  │──── зашифровано ─────→│  Сервер  │
└──────────┘                      └──────────┘
        ↑
   [Хакер бачить тільки шифр]
```

### Ключові характеристики TLS:

1. **Працює на транспортному рівні** — між TCP та HTTP (рівень 5-6 моделі OSI)
2. **Використовує два типи шифрування** — асиметричне (для обміну ключами) та симетричне (для даних)
3. **Двостороння перевірка** — сервер та/або клієнт можуть автентифікуватись
4. **Прозорий для додатків** — HTTP не знає про TLS, просто передає дані

## Історія та еволюція

### SSL 1.0 (1994) — Ніколи не випущений

-   Розроблений Netscape
-   Мав критичні вразливості
-   Ніколи не був публічно випущений

### SSL 2.0 (1995) — Перша публічна версія

**Характеристики:**
-   Підтримка базового шифрування
-   Автентифікація сервера

**Проблеми:**
-   Вразливості в дизайні
-   Слабкі криптографічні алгоритми
-   Вже застарілий і небезпечний ⚠️

### SSL 3.0 (1996) — Повне перероблення

**Покращення:**
-   Виправлено вразливості SSL 2.0
-   Краща криптографічна модель
-   Підтримка більше cipher suites

**Статус:** Офіційно застарілий у 2015 через POODLE атаку ⚠️

### TLS 1.0 (1999) — Новий стандарт

**Що нового:**
-   Стандартизований IETF (RFC 2246)
-   Несумісний з SSL 3.0 (хоча схожий)
-   Кращі алгоритми хешування (HMAC)

**Статус:** Застарілий у 2020 ⚠️

### TLS 1.1 (2006) — Захист від атак

**Покращення:**
-   Захист від CBC атак
-   Явні Initialization Vectors (IV)
-   Підтримка більше cipher suites

**Статус:** Застарілий у 2020 ⚠️

### TLS 1.2 (2008) — Сучасний стандарт

**Ключові зміни:**
-   Підтримка SHA-256 та інших сучасних хеш-функцій
-   Можливість вказувати hash алгоритм для підпису
-   Authenticated Encryption (AES-GCM, ChaCha20-Poly1305)
-   Захист від багатьох відомих атак

**Статус:** ✅ Рекомендований (2024)

### TLS 1.3 (2018) — Новітній стандарт

**Революційні зміни:**
-   ⚡ Швидший handshake (1-RTT замість 2-RTT)
-   🔒 0-RTT режим (ще швидше, але з обмеженнями)
-   🗑️ Видалено застарілі cipher suites
-   🛡️ Завжди шифрує handshake
-   📉 Менше вразливостей через простіший дизайн

**Статус:** ✅ Найкращий вибір (2024)

### Хронологія:

```
1994 ─ SSL 1.0 (не випущений)
1995 ─ SSL 2.0 ⚠️ застарілий
1996 ─ SSL 3.0 ⚠️ застарілий (POODLE)
1999 ─ TLS 1.0 ⚠️ застарілий з 2020
2006 ─ TLS 1.1 ⚠️ застарілий з 2020
2008 ─ TLS 1.2 ✅ ще використовується
2018 ─ TLS 1.3 ✅ рекомендується
```

### Підтримка браузерами (2024):

| Браузер | TLS 1.2 | TLS 1.3 |
|---------|---------|---------|
| Chrome 120+ | ✅ | ✅ |
| Firefox 120+ | ✅ | ✅ |
| Safari 17+ | ✅ | ✅ |
| Edge 120+ | ✅ | ✅ |

**Важливо:** Більшість сучасних браузерів ВІДКЛЮЧИЛИ підтримку SSL 2.0, SSL 3.0, TLS 1.0 та TLS 1.1!

## Навіщо потрібне шифрування?

### Загрози без шифрування:

#### 1. **Перехоплення (Eavesdropping)**

```
☕ Ви в кав'ярні, підключились до Wi-Fi

Ваш комп'ютер → [Wi-Fi роутер] → Інтернет
                      ↑
                 [Хакер слухає]

Без HTTPS:
- Логіни та паролі ❌
- Номери кредиток ❌
- Особисті повідомлення ❌
- Cookies і сесії ❌

З HTTPS:
- Все зашифровано ✅
```

**Реальний приклад:**
```
HTTP запит (видимий хакеру):
POST /login HTTP/1.1
Host: example.com

username=ivan&password=mypassword123

→ Хакер бачить пароль!
```

#### 2. **Підміна даних (Man-in-the-Middle)**

```
Атака Man-in-the-Middle (MITM):

Ви думаєте:
Клієнт ←→ bank.com

Насправді:
Клієнт ←→ [ХАКЕР] ←→ bank.com

Хакер може:
- Читати всі дані
- Змінювати суми транзакцій
- Красти паролі
```

**Приклад атаки:**
```
Ви: "Переказати 100 грн на рахунок 12345"
     ↓
Хакер змінює: "Переказати 10000 грн на рахунок ХАКЕРА"
     ↓
Банк отримує підроблений запит
```

#### 3. **Фішинг (підробка сайтів)**

```
Без TLS:
http://bank.com ← справжній
http://baпk.com ← підробка (кирилиця 'а')

Користувач не помітить різницю! ❌

З TLS:
https://bank.com 🔒 ← сертифікат перевірений ✅
https://baпk.com ⚠️ ← немає сертифіката або невірний ❌
```

#### 4. **Session Hijacking**

```
Без HTTPS:
Cookie: sessionId=abc123

Хакер перехоплює cookie → заходить під вашим акаунтом ❌

З HTTPS + Secure cookie:
Cookie: sessionId=abc123; Secure; HttpOnly

Cookie зашифрований у TLS тунелі ✅
```

### Що захищає TLS:

```
Незахищений HTTP запит:
───────────────────────────────────────
GET /account HTTP/1.1
Host: bank.com
Cookie: session=abc123
Authorization: Bearer secret_token

{"transfer": 1000, "to": "12345"}
───────────────────────────────────────
↑ ВСЕ видно як є! ❌


TLS зашифрований:
───────────────────────────────────────
A8 3F 2D 9B 4E 7C 1F 8A 3D 6B 2E 9F 4A
7B 5D 3C 9A 2F 8E 1D 6C 4B 7A 9E 3F 2D
8C 1A 5F 7E 3B 9D 2C 6A 4F 8B 1E 7D 3A
───────────────────────────────────────
↑ Хакер бачить тільки "білий шум" ✅
```

### Приклади реальних атак (історичні):

#### Firesheep (2010)
```
- Розширення для Firefox
- Перехоплювало незашифровані cookies у Wi-Fi мережах
- Дозволяло "красти" сесії Facebook, Twitter
- Змусило сайти перейти на HTTPS
```

#### SSL Strip (2009)
```
- Атака MITM
- Перехоплювач замінював HTTPS на HTTP
- Користувач не помічав (немає 🔒 замочка)
- Вирішення: HSTS (HTTP Strict Transport Security)
```

## Основи криптографії

**Криптографія** — наука про захист інформації за допомогою математичних перетворень.

### Основні поняття:

#### Plaintext (відкритий текст)
```
Оригінальне повідомлення, яке потрібно захистити

Приклад: "Привіт, Марія!"
```

#### Ciphertext (шифротекст)
```
Зашифроване повідомлення

Приклад: "X8mK@3pL$9qR#..."
```

#### Encryption (шифрування)
```
Процес перетворення plaintext → ciphertext

Plaintext + Ключ → [Алгоритм шифрування] → Ciphertext
```

#### Decryption (розшифрування)
```
Зворотний процес: ciphertext → plaintext

Ciphertext + Ключ → [Алгоритм розшифрування] → Plaintext
```

#### Key (ключ)
```
Секретна інформація, необхідна для шифрування/розшифрування

Приклади:
- Пароль
- Випадкове число
- Файл із секретними даними
```

### Візуалізація процесу:

```
ШИФРУВАННЯ:
┌────────────┐         ┌──────────────┐         ┌─────────────┐
│  Plaintext │────────→│  Encryption  │────────→│ Ciphertext  │
│ "Привіт!"  │         │  Algorithm   │         │ "X8mK@3pL"  │
└────────────┘         └──────────────┘         └─────────────┘
                              ↑
                          [Ключ]

РОЗШИФРУВАННЯ:
┌─────────────┐        ┌──────────────┐         ┌────────────┐
│ Ciphertext  │───────→│  Decryption  │────────→│  Plaintext │
│ "X8mK@3pL"  │        │  Algorithm   │         │ "Привіт!"  │
└─────────────┘        └──────────────┘         └────────────┘
                              ↑
                          [Ключ]
```

### Два основні типи шифрування:

#### 1. Симетрична криптографія
```
Один ключ для шифрування ТА розшифрування

    [Один ключ]
         ↓
Шифрування ←→ Розшифрування

Приклади: AES, ChaCha20, 3DES
```

#### 2. Асиметрична криптографія
```
Пара ключів: публічний + приватний

Публічний ключ → Шифрування
Приватний ключ → Розшифрування

Приклади: RSA, ECDSA, EdDSA
```

### Принцип Кергофса (Kerckhoffs's principle):

> **"Безпека системи повинна залежати тільки від секретності ключа, а не від секретності алгоритму"**

```
❌ Погано:
"Ніхто не знає мій секретний алгоритм шифрування"
→ Якщо хтось дізнається алгоритм, все зламано!

✅ Добре:
"Використовую AES (всім відомий алгоритм), але тільки я знаю ключ"
→ Навіть якщо всі знають алгоритм, без ключа нічого не зроблять!
```

## Симетрична криптографія

**Симетрична криптографія** використовує **один і той самий ключ** для шифрування та розшифрування.

### Як це працює:

```
Аліса хоче відправити повідомлення Богдану

1. Аліса і Богдан домовляються про секретний ключ
   Ключ: "my-secret-key-123"

2. Аліса шифрує повідомлення
   "Привіт!" + Ключ → "X8mK@3pL"

3. Аліса відправляє шифротекст Богдану
   "X8mK@3pL" ───→ Богдан

4. Богдан розшифровує ТИМ САМИМ ключем
   "X8mK@3pL" + Ключ → "Привіт!"
```

### Візуалізація:

```
      [Секретний ключ]
            ↓ ↑
Аліса ──→ 🔐 ──→ Богдан
      шифрує   розшифровує
```

### Переваги симетричного шифрування:

✅ **Дуже швидке** — тисячі мегабайтів за секунду
✅ **Ефективне** — малі ключі (128-256 біт)
✅ **Просте** — менше обчислень

### Недоліки:

❌ **Проблема обміну ключами** — як безпечно передати ключ?
❌ **Багато ключів** — для N користувачів потрібно N×(N-1)/2 ключів

```
Приклад проблеми:

3 користувачі:
Аліса ←→ Богдан (ключ 1)
Аліса ←→ Віктор (ключ 2)
Богдан ←→ Віктор (ключ 3)
= 3 ключі

10 користувачів:
= 45 ключів!

1000 користувачів:
= 499500 ключів! 😱
```

### Популярні алгоритми:

#### AES (Advanced Encryption Standard)

```
Найпопулярніший сучасний алгоритм

Розміри ключів:
- AES-128: 128 біт (16 байтів)
- AES-192: 192 біти (24 байти)
- AES-256: 256 біт (32 байти)

Швидкість: Дуже висока
Безпека: Відмінна
Статус: ✅ Рекомендований
```

**Як працює AES (спрощено):**

```
Plaintext розділяється на блоки по 128 біт (16 байтів)

Кожен блок проходить через раунди перетворень:

Раунд:
1. SubBytes    — заміна байтів (нелінійна операція)
2. ShiftRows   — зсув рядків
3. MixColumns  — перемішування стовпців
4. AddRoundKey — XOR з ключем раунду

AES-128: 10 раундів
AES-192: 12 раундів
AES-256: 14 раундів

Результат: Ciphertext
```

**Приклад (концептуальний):**

```
Plaintext: "Hello World!!!!!" (16 байтів)

┌─────────────────┐
│ H e l l o   W o │
│ r l d ! ! ! ! ! │
└─────────────────┘
        ↓
   [10 раундів AES-128]
        ↓
┌─────────────────┐
│ 8F 3A D7 2B E4  │ ← шифротекст (виглядає як випадкові дані)
│ 91 5C 6A F2 D8  │
└─────────────────┘
```

#### ChaCha20

```
Сучасний потоковий шифр від Daniel J. Bernstein

Розмір ключа: 256 біт
Швидкість: Дуже висока (особливо на мобільних)
Безпека: Відмінна
Статус: ✅ Рекомендований (особливо для мобільних пристроїв)

Використовується в:
- TLS (ChaCha20-Poly1305)
- WireGuard VPN
- Android шифрування дисків
```

#### 3DES (Triple DES)

```
Застарілий алгоритм (потрійне застосування DES)

Розмір ключа: 168 біт (3 × 56 біт)
Швидкість: Повільна
Безпека: Слабка за сучасними стандартами
Статус: ⚠️ Застарілий, не використовувати!
```

### Режими роботи блокових шифрів:

Блокові шифри (AES) шифрують фіксовані блоки даних. Але що робити, якщо дані більші за 1 блок?

#### ECB (Electronic Codebook) — ❌ НЕБЕЗПЕЧНИЙ

```
Кожен блок шифрується незалежно

Plaintext:  [Block 1] [Block 2] [Block 3]
               ↓          ↓          ↓
            [Encrypt] [Encrypt] [Encrypt]
               ↓          ↓          ↓
Ciphertext: [Block 1] [Block 2] [Block 3]

Проблема: Однакові блоки plaintext → однакові блоки ciphertext
```

**Приклад проблеми ECB:**

```
Зображення пінгвіна, зашифроване ECB:
- Контури видно!
- Patterns зберігаються!
- НЕБЕЗПЕЧНО! ❌
```

#### CBC (Cipher Block Chaining) — ⚠️ Обережно

```
Кожен блок XOR'иться з попереднім шифротекстом

Plaintext:  [Block 1]     [Block 2]     [Block 3]
               ⊕              ⊕              ⊕
              [IV]       [Cipher 1]    [Cipher 2]
               ↓              ↓              ↓
            [Encrypt]     [Encrypt]     [Encrypt]
               ↓              ↓              ↓
Ciphertext: [Block 1]     [Block 2]     [Block 3]

IV (Initialization Vector) — випадкове значення для першого блоку
```

**Недолік:** Вразливий до padding oracle атак, потребує MAC окремо.

#### GCM (Galois/Counter Mode) — ✅ Рекомендований

```
Режим з вбудованою автентифікацією (AEAD)

Переваги:
✅ Швидкий (можна паралелізувати)
✅ Автентифікація вбудована (не потрібен окремий MAC)
✅ Захист від підміни даних

Використовується:
- AES-GCM у TLS 1.2/1.3
- IPsec
- Шифрування дисків
```

**Приклад використання:**

```
AES-256-GCM шифрування:

Input:
- Plaintext: "Секретне повідомлення"
- Key: 256-bit ключ
- Nonce: унікальне число (96 біт)
- Additional Data: "header info" (опціонально)

Output:
- Ciphertext: зашифровані дані
- Authentication Tag: 128-bit тег для перевірки цілісності

При розшифруванні:
1. Перевіряється Authentication Tag
2. Якщо тег невірний → дані підроблені! ❌
3. Якщо тег вірний → розшифровуємо ✅
```

### Проблема обміну ключами:

```
Аліса і Богдан хочуть спілкуватись безпечно

Проблема: Як передати секретний ключ?

❌ Відправити email: незахищено!
❌ Телефонувати: можуть прослуховувати!
❌ Передати особисто: незручно, якщо на різних континентах!

Рішення: Асиметрична криптографія! →
```

## Асиметрична криптографія

**Асиметрична криптографія** (або **криптографія з відкритим ключем**) використовує **пару ключів**:
- **Публічний ключ** (Public Key) — можна вільно розповсюджувати
- **Приватний ключ** (Private Key) — зберігається в секреті

### Магія асиметричного шифрування:

```
Що зашифровано публічним ключем → можна розшифрувати ТІЛЬКИ приватним
Що зашифровано приватним ключем → можна розшифрувати публічним
```

### Як це працює:

```
1. Богдан генерує пару ключів:
   🔑 Приватний ключ (зберігає в секреті)
   🔓 Публічний ключ (публікує відкрито)

2. Аліса хоче надіслати секретне повідомлення Богдану:
   - Аліса бере ПУБЛІЧНИЙ ключ Богдана (доступний усім)
   - Шифрує повідомлення: "Привіт!" + 🔓 → "X8mK@3pL"
   - Відправляє шифротекст Богдану

3. Богдан отримує шифротекст:
   - Використовує свій ПРИВАТНИЙ ключ
   - Розшифровує: "X8mK@3pL" + 🔑 → "Привіт!"

4. ✅ Ніхто інший не може розшифрувати, навіть маючи публічний ключ!
```

### Візуалізація:

```
Аліса                           Богдан
  │                               │
  │  1. Запитує публічний ключ   │
  │ ──────────────────────────→  │
  │                               │
  │  2. Публічний ключ 🔓        │
  │ ←──────────────────────────  │
  │                               │
  │  3. Шифрує повідомлення      │
  │     "Привіт" + 🔓 = шифр     │
  │                               │
  │  4. Відправляє шифротекст    │
  │ ──────────────────────────→  │
  │                               │
  │                           5. Розшифровує
  │                              шифр + 🔑 = "Привіт"
```

### Математична основа (спрощено):

Асиметрична криптографія базується на **односторонніх функціях** — легко обчислити в одному напрямку, але дуже важко у зворотному.

**Приклад односторонньої функції:**
```
Легко: 127 × 131 = 16,637 ✅ (швидко)
Важко: Які два прості числа дають 16,637? ❌ (потрібен час)

Для великих чисел (2048 біт):
Легко: помножити два 1024-бітних числа = мілісекунди
Важко: розкласти 2048-бітне число = тисячі років обчислень!
```

### Популярні алгоритми:

#### RSA (Rivest–Shamir–Adleman)

```
Найстаріший і найвідоміший алгоритм асиметричного шифрування

Основа: Складність факторизації великих чисел

Розміри ключів:
- RSA-2048: 2048 біт (рекомендований мінімум)
- RSA-3072: 3072 біти (краще)
- RSA-4096: 4096 біт (максимальна безпека)

Швидкість: Повільний (особливо порівняно з симетричними алгоритмами)
Безпека: Хороша (якщо ключ достатньо великий)
Статус: ✅ Досі використовується, але ECDSA краще
```

**Як працює RSA (дуже спрощено):**

```
Генерація ключів:
1. Вибрати два великі прості числа: p і q
2. Обчислити n = p × q
3. Обчислити φ(n) = (p-1)(q-1)
4. Вибрати e (публічний експонент), зазвичай 65537
5. Обчислити d (приватний експонент): d × e ≡ 1 (mod φ(n))

Публічний ключ: (n, e)
Приватний ключ: (n, d)

Шифрування:
ciphertext = plaintext^e mod n

Розшифрування:
plaintext = ciphertext^d mod n
```

**Приклад з маленькими числами:**

```
(У реальності числа мають сотні цифр!)

1. Вибираємо p=61, q=53
2. n = 61 × 53 = 3233
3. φ(n) = 60 × 52 = 3120
4. e = 17 (публічний експонент)
5. d = 2753 (приватний експонент, d × 17 ≡ 1 mod 3120)

Публічний ключ: (3233, 17)
Приватний ключ: (3233, 2753)

Шифруємо повідомлення M=123:
C = 123^17 mod 3233 = 855

Розшифровуємо:
M = 855^2753 mod 3233 = 123 ✅
```

#### ECDSA (Elliptic Curve Digital Signature Algorithm)

```
Сучасний алгоритм на основі еліптичних кривих

Переваги:
✅ Менші ключі (256 біт ECDSA ≈ 3072 біт RSA за безпекою)
✅ Швидше
✅ Менше пам'яті та обчислень

Розміри ключів:
- P-256 (secp256r1): 256 біт
- P-384 (secp384r1): 384 біти
- P-521 (secp521r1): 521 біт

Швидкість: Швидше за RSA
Безпека: Відмінна
Статус: ✅ Рекомендований для нових систем
```

**Як працює ECDSA (концептуально):**

```
Еліптична крива — це рівняння виду:
y² = x³ + ax + b

На цій кривій визначена операція "додавання точок"

Приватний ключ: випадкове число d
Публічний ключ: точка Q = d × G (G — базова точка на кривій)

Обчислити Q з d — легко
Обчислити d з Q — практично неможливо! (Discrete Logarithm Problem)
```

**Візуалізація еліптичної кривої:**

```
      y
      │
      │       •
      │     •   •
      │    •     •
──────┼────•──────•──────── x
      │   •       •
      │   •     •
      │    •   •
      │      •
```

#### EdDSA (Edwards-curve Digital Signature Algorithm)

```
Новітній алгоритм на основі кривих Едвардса

Варіанти:
- Ed25519: 256 біт (найпопулярніший)
- Ed448: 448 біт

Переваги:
✅ Ще швидше за ECDSA
✅ Безпечніша реалізація (менше можливостей для помилок)
✅ Детермінований (немає потреби у випадковості при підписі)

Використовується в:
- SSH ключі
- TLS 1.3
- Signal, WhatsApp
- Cryptocurrencies (Bitcoin, Ethereum використовують ECDSA, але нові проєкти часто обирають EdDSA)

Статус: ✅ Найкращий вибір для нових систем
```

### Порівняння розмірів ключів (еквівалентна безпека):

| Симетричний | RSA    | ECDSA/EdDSA |
|-------------|--------|-------------|
| 80 біт      | 1024   | 160         |
| 112 біт     | 2048   | 224         |
| 128 біт     | 3072   | 256         |
| 192 біт     | 7680   | 384         |
| 256 біт     | 15360  | 521         |

**Висновок:** ECDSA/EdDSA набагато ефективніші!

### Переваги асиметричного шифрування:

✅ **Вирішує проблему обміну ключами** — публічний ключ можна передавати відкрито
✅ **Масштабованість** — N користувачів = N пар ключів (не N²!)
✅ **Цифрові підписи** — можна підтверджувати автентичність

### Недоліки:

❌ **Повільне** — в 100-1000 разів повільніше за симетричне шифрування
❌ **Великі ключі** — RSA-2048 = 2048 біт, AES-256 = 256 біт
❌ **Складніше** — більше обчислень

### Чому TLS використовує ОБА типи шифрування?

```
🎯 Оптимальне рішення: Гібридна криптографія

1. Асиметричне шифрування (RSA/ECDSA):
   - Використовується ТІЛЬКИ для обміну ключами
   - Повільно, але вирішує проблему безпечного обміну

2. Симетричне шифрування (AES):
   - Використовується для шифрування даних
   - Швидко, ефективно

Процес TLS:
┌─────────────────────────────────────┐
│ 1. Handshake (асиметричне)          │
│    → Обмін ключами                  │
│    → Автентифікація сертифікатів    │
│    Результат: Session Key           │
└─────────────────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ 2. Передача даних (симетричне)      │
│    → Шифрування/розшифрування AES   │
│    → Використовується Session Key   │
│    → Швидко і ефективно ✅          │
└─────────────────────────────────────┘
```

### Приклад використання:

```
Scenario: Аліса відкриває https://bank.com

1. Аліса ← (публічний ключ банку)

2. Аліса генерує випадковий ключ: "random_key_123"

3. Аліса шифрує ключ публічним ключем банку:
   "random_key_123" + 🔓 банку → "X8mK@3pL..."

4. Аліса → банк: "X8mK@3pL..."

5. Банк розшифровує приватним ключем:
   "X8mK@3pL..." + 🔑 банку → "random_key_123"

6. ✅ Тепер обидві сторони мають "random_key_123"!

7. Подальша комунікація використовує AES з "random_key_123":
   - Швидко
   - Безпечно
   - Ніхто інший не знає ключа
```

## Хеш-функції

**Хеш-функція** — це математична функція, яка перетворює дані будь-якого розміру у фіксований розмір (хеш).

### Властивості хеш-функції:

#### 1. **Детермінізм**
```
Один і той самий вхід → завжди той самий хеш

"Привіт" → a3f8b9c... (завжди)
"Привіт" → a3f8b9c... (знову той самий)
```

#### 2. **Швидкість обчислення**
```
Легко і швидко обчислити хеш для будь-яких даних
```

#### 3. **Односторонність (One-way)**
```
Легко: Дані → Хеш ✅
Неможливо: Хеш → Дані ❌

"password123" → 482c811da5d5b4bc6d497ffa98491e38
Неможливо відновити "password123" з хешу!
```

#### 4. **Лавинний ефект (Avalanche Effect)**
```
Маленька зміна у вхідних даних → повністю інший хеш

"Привіт"  → a3f8b9c2d1e4f7a8b3c6d9e2f5a8b1c4
"Привіт!" → 8e2f5a8b1c4d7e0a3f6b9c2e5d8a1b4f
            ↑ зовсім інший!
```

#### 5. **Collision Resistance (стійкість до колізій)**
```
Практично неможливо знайти два різні входи з однаковим хешем

"Привіт" → a3f8b9c...
"Hello"  → 8e2f5a8... (різні!)

Колізія: "Привіт" → a3f8b9c...
         "Хай"    → a3f8b9c... (той самий хеш) ← це ПОГАНО!

Хороша хеш-функція: колізії практично неможливі
```

### Візуалізація:

```
Будь-які дані → [Хеш-функція] → Фіксований розмір

"A"                    → a3f8b9c2d1e4f7a8...
"Привіт"               → 8e2f5a8b1c4d7e0a...
Весь текст Війни і Миру → 1b4f7a8c3e6d9f2a...
Відео 4K (10 GB)       → 5a8b1c4e7d0f3a6b...
                          
                          ↑ Завжди той самий розмір!
```

### Популярні хеш-функції:

#### MD5 (Message Digest 5)

```
Створений у 1991

Розмір хешу: 128 біт (32 hex символи)
Швидкість: Дуже швидка
Безпека: ❌ ЗЛАМАНА! Легко знайти колізії!
Статус: ⚠️ НЕ використовувати для безпеки!

Можна використовувати тільки для:
- Checksum (перевірка цілісності файлів, не для безпеки)
- Non-cryptographic purposes
```

**Приклад MD5:**
```
"Привіт" → f5e7d2b1c3a4f6e8d9a2b5c7e1f3a8b2

Проблема: колізії можна знайти за секунди!
```

#### SHA-1 (Secure Hash Algorithm 1)

```
Створений у 1995 (NSA)

Розмір хешу: 160 біт (40 hex символів)
Швидкість: Швидка
Безпека: ⚠️ Вразлива! Колізії знайдені у 2017!
Статус: ⚠️ Застаріла, не використовувати для нових систем!
```

**Приклад SHA-1:**
```
"Привіт" → 7f8b3c2d4e5f1a6b9c8e7d2a5f3b8c1e4a6f9d2c
```

**SHAttered атака (2017):**
```
Google знайшов колізію SHA-1:
Два різні PDF файли з однаковим SHA-1 хешем!
→ SHA-1 більше не безпечна для криптографії
```

#### SHA-2 (SHA-256, SHA-384, SHA-512)

```
Сімейство сучасних хеш-функцій

SHA-256:
- Розмір: 256 біт (64 hex символи)
- Швидкість: Швидка
- Безпека: ✅ Відмінна
- Використання: Bitcoin, TLS, сертифікати

SHA-384:
- Розмір: 384 біти
- Безпека: ✅ Ще краще

SHA-512:
- Розмір: 512 біт
- Швидкість: Швидша на 64-бітних системах
- Безпека: ✅ Максимальна

Статус: ✅ Рекомендовано (2024)
```

**Приклад SHA-256:**
```
"Привіт" → 3f8a2c7e9b1d4f6a8c3e5d7b9f1a4c6e8d2b5f7a9c3e6d1f4a8b2c5e7d9f3a6c

"Привіт!" → e7d2a5f8c1b4e6a9d3f7c2e5a8d1b4f7c9e2a6d3f8b1c4e7d9a2f5c8e1b4d6
             ↑ Повністю інший!
```

#### SHA-3

```
Новітній стандарт (2015)

Базується на іншій математичній основі (Keccak)

Розміри: SHA3-256, SHA3-384, SHA3-512
Швидкість: Трохи повільніше за SHA-2
Безпека: ✅ Відмінна
Статус: ✅ Хороший вибір, але SHA-2 більш поширений
```

### Використання хеш-функцій у TLS:

#### 1. **Перевірка цілісності повідомлень (HMAC)**

```
HMAC = Hash-based Message Authentication Code

HMAC(key, message) → tag

Перевірка:
1. Відправник обчислює: HMAC("secret", "Привіт") → abc123...
2. Відправляє: "Привіт" + "abc123..."
3. Отримувач обчислює: HMAC("secret", "Привіт") → abc123...
4. Порівнює теги: abc123 == abc123 ✅ Дані не змінені!

Якщо хакер змінить повідомлення:
"Привіт" → "Хай"
HMAC("secret", "Хай") → xyz789... ≠ abc123... ❌ Виявлено підміну!
```

#### 2. **Підписи сертифікатів**

```
Сертифікат підписується хешем:
1. Хеш(сертифікат) → hash_value
2. Підпис(hash_value, private_key) → signature

Перевірка:
1. Хеш(сертифікат) → hash_value
2. Перевірити підпис з публічним ключем ✅
```

#### 3. **Fingerprint сертифікатів**

```
SHA-256 fingerprint сертифіката:
SHA-256(сертифікат) → A3:F8:B9:C2:D1:E4:...

Використовується для:
- Ідентифікації сертифікатів
- Certificate pinning
```

#### 4. **Виведення ключів (Key Derivation)**

```
З одного master secret можна вивести багато ключів:

master_secret → [HKDF-SHA256] → {
  client_write_key,
  server_write_key,
  client_IV,
  server_IV
}
```

### Приклад роботи хеш-функції:

```python
# SHA-256 приклад (концептуально)

import hashlib

# Хеш рядка
text = "Привіт"
hash1 = hashlib.sha256(text.encode()).hexdigest()
print(hash1)  # 3f8a2c7e9b1d4f6a8c3e5d7b9f1a4c6e8d2b5f7a9c3e6d1f4a8b2c5e7d9f3a6c

# Той самий текст → той самий хеш
hash2 = hashlib.sha256(text.encode()).hexdigest()
print(hash1 == hash2)  # True ✅

# Маленька зміна → повністю інший хеш
text2 = "Привіт!"
hash3 = hashlib.sha256(text2.encode()).hexdigest()
print(hash3)  # e7d2a5f8c1b4e6a9d3f7c2e5a8d1b4f7c9e2a6d3f8b1c4e7d9a2f5c8e1b4d6
print(hash1 == hash3)  # False ❌ (повністю різні)
```

## Цифрові підписи

**Цифровий підпис** — це криптографічний механізм, який дозволяє підтвердити автентичність та цілісність повідомлення.

### Аналогія з ручним підписом:

```
Ручний підпис на папері:
- Підтверджує, що ВИ підписали документ
- Важко підробити
- Якщо документ змінено → підпис недійсний

Цифровий підпис:
- Підтверджує автора (автентифікація)
- Неможливо підробити без приватного ключа
- Якщо дані змінено → підпис недійсний
- + можна перевірити автоматично!
```

### Як працює цифровий підпис:

```
Підписання:
1. Обчислити хеш повідомлення
2. Зашифрувати хеш приватним ключем = підпис

Перевірка:
1. Обчислити хеш отриманого повідомлення
2. Розшифрувати підпис публічним ключем = оригінальний хеш
3. Порівняти хеші: однакові = підпис валідний ✅
```

### Детальний процес:

```
ПІДПИСАННЯ (Аліса):

1. Аліса має повідомлення: "Переказати 100 грн Богдану"

2. Обчислюємо хеш:
   SHA-256("Переказати 100 грн Богдану") → a3f8b9c2d1e4...

3. Підписуємо хеш приватним ключем:
   Підпис = Encrypt(a3f8b9c2d1e4..., приватний_ключ_Аліси)
   → x8m3k9p2l7...

4. Відправляємо:
   Повідомлення: "Переказати 100 грн Богдану"
   Підпис: x8m3k9p2l7...
   Публічний ключ: (щоб інші могли перевірити)


ПЕРЕВІРКА (Богдан):

1. Богдан отримує:
   - Повідомлення: "Переказати 100 грн Богдану"
   - Підпис: x8m3k9p2l7...
   - Публічний ключ Аліси

2. Обчислюємо хеш отриманого повідомлення:
   SHA-256("Переказати 100 грн Богдану") → a3f8b9c2d1e4...

3. Розшифровуємо підпис публічним ключем Аліси:
   Decrypt(x8m3k9p2l7..., публічний_ключ_Аліси) → a3f8b9c2d1e4...

4. Порівнюємо:
   Хеш з пункту 2: a3f8b9c2d1e4...
   Хеш з підпису:  a3f8b9c2d1e4...
   
   Однакові → ✅ Підпис валідний!
   Це справді Аліса, і повідомлення не змінено!
```

### Візуалізація:

```
ПІДПИСАННЯ:
                          [Приватний ключ 🔑]
                                   ↓
Повідомлення → [Hash] → Хеш → [Encrypt] → Підпис
     │                                        │
     └────────────────────────────────────────┴──→ Відправлено


ПЕРЕВІРКА:
                          [Публічний ключ 🔓]
                                   ↓
               Підпис → [Decrypt] → Хеш₁
                                     │
Повідомлення → [Hash] → Хеш₂        │
                         │           │
                         └─[Compare]─┘
                              │
                         ✅ Однакові?
```

### Що гарантує цифровий підпис:

#### 1. **Автентифікація**
```
Підпис створений приватним ключем Аліси
→ Тільки Аліса могла створити цей підпис
→ ✅ Це справді від Аліси
```

#### 2. **Цілісність**
```
Якщо хтось змінить повідомлення:
"Переказати 100 грн" → "Переказати 1000 грн"

Новий хеш: b9e2f5c8...
Хеш з підпису: a3f8b9c2...

b9e2f5c8 ≠ a3f8b9c2 → ❌ Підпис недійсний!
→ Дані було змінено!
```

#### 3. **Невідмовність (Non-repudiation)**
```
Аліса не може сказати "це не я підписала"
→ Тільки вона має приватний ключ
→ ✅ Вона дійсно підписала
```

### Використання у TLS:

#### 1. **Підпис сертифікатів**

```
Certificate Authority (CA) підписує сертифікат:

1. CA обчислює хеш сертифіката
2. Підписує хеш своїм приватним ключем
3. Додає підпис до сертифіката

Браузер перевіряє:
1. Обчислює хеш сертифіката
2. Розшифровує підпис публічним ключем CA
3. Порівнює → ✅ Сертифікат валідний!
```

#### 2. **Автентифікація сервера**

```
Під час TLS handshake:

Сервер доводить, що володіє приватним ключем:
1. Клієнт відправляє випадкові дані
2. Сервер підписує їх своїм приватним ключем
3. Клієнт перевіряє підпис з публічного ключа (з сертифіката)
4. ✅ Це справді той сервер!
```

### Алгоритми цифрових підписів:

#### RSA signatures
```
Використовує RSA для створення підпису

Розміри: 2048, 3072, 4096 біт
Швидкість: Повільно
Безпека: Хороша
Статус: ✅ Досі використовується
```

#### ECDSA (Elliptic Curve Digital Signature Algorithm)
```
Підписи на основі еліптичних кривих

Розміри: P-256, P-384, P-521
Швидкість: Швидше за RSA
Розмір підпису: Менший
Безпека: Відмінна
Статус: ✅ Рекомендовано
```

#### EdDSA (Ed25519)
```
Найсучасніший алгоритм

Розмір: 256 біт
Швидкість: Найшвидший
Розмір підпису: 64 байти (дуже компактний!)
Безпека: Найкраща
Статус: ✅ Ідеальний для нових систем
```

### Порівняння:

| Алгоритм | Розмір ключа | Розмір підпису | Швидкість | Безпека |
|----------|-------------|----------------|-----------|---------|
| RSA-2048 | 2048 біт    | 256 байт       | Повільно  | ✅      |
| ECDSA P-256 | 256 біт  | 64 байти       | Швидко    | ✅      |
| Ed25519  | 256 біт     | 64 байти       | Найшвидше | ✅✅    |

### Практичний приклад:

```bash
# Генерація RSA ключів
openssl genrsa -out private.pem 2048
openssl rsa -in private.pem -pubout -out public.pem

# Підписання файлу
echo "Важливе повідомлення" > message.txt
openssl dgst -sha256 -sign private.pem -out signature.bin message.txt

# Перевірка підпису
openssl dgst -sha256 -verify public.pem -signature signature.bin message.txt
# Результат: Verified OK ✅

# Якщо змінити файл:
echo "Змінене повідомлення" > message.txt
openssl dgst -sha256 -verify public.pem -signature signature.bin message.txt
# Результат: Verification Failure ❌
```

### Атака: підміна відкритого ключа

```
Проблема: як бути впевненим, що публічний ключ справді належить Алісі?

Сценарій MITM атаки:

1. Аліса хоче отримати публічний ключ Богдана
2. Аліса → (запит на публічний ключ) → Хакер → Богдан
3. Богдан → (свій публічний ключ) → Хакер (ПЕРЕХОПЛЮЄ!)
4. Хакер → (публічний ключ ХАКЕРА!) → Аліса
5. Аліса думає, що це ключ Богдана, але насправді це ключ хакера!

Вирішення: Сертифікати та PKI (Public Key Infrastructure) →
```

## Сертифікати та PKI

### Що таке цифровий сертифікат?

**Цифровий сертифікат** (або **SSL/TLS сертифікат**) — це електронний документ, який прив'язує публічний ключ до ідентичності власника (домен, організація, особа).

**Аналогія:** Як паспорт для веб-сайту.

```
Паспорт:
- Фото
- Ім'я
- Дата народження
- Підпис уряду (довіряємо уряду)

Цифровий сертифікат:
- Публічний ключ
- Доменне ім'я (example.com)
- Власник (компанія)
- Підпис Certificate Authority (довіряємо CA)
```

### Структура сертифіката (X.509)

```
┌─────────────────────────────────────────┐
│ СЕРТИФІКАТ                              │
├─────────────────────────────────────────┤
│ Версія: v3                              │
│ Серійний номер: 1234567890ABCDEF        │
│                                         │
│ ВЛАСНИК (Subject):                      │
│   Common Name: example.com              │
│   Organization: Example Inc.            │
│   Country: UA                           │
│                                         │
│ ЕМІТЕНТ (Issuer):                       │
│   Common Name: Let's Encrypt Authority  │
│   Organization: Let's Encrypt           │
│                                         │
│ ЧАС ДІЇ:                                │
│   Від: 2024-01-01 00:00:00 UTC          │
│   До:  2024-12-31 23:59:59 UTC          │
│                                         │
│ ПУБЛІЧНИЙ КЛЮЧ:                         │
│   Алгоритм: RSA 2048 біт                │
│   Модуль: 00:C5:3A:... (2048 біт)       │
│   Експонент: 65537                      │
│                                         │
│ РОЗШИРЕННЯ:                             │
│   Subject Alternative Names:            │
│     - example.com                       │
│     - www.example.com                   │
│     - *.example.com                     │
│   Key Usage: Digital Signature          │
│   Extended Key Usage: TLS Server Auth   │
│                                         │
│ ЦИФРОВИЙ ПІДПИС CA:                     │
│   Алгоритм: SHA-256 with RSA            │
│   Підпис: A3:F8:B9:... (256 байт)       │
└─────────────────────────────────────────┘
```

### Основні поля сертифіката:

#### 1. **Subject (Власник)**
```
CN (Common Name): example.com
O (Organization): Example Corporation
OU (Organizational Unit): IT Department
L (Locality): Kyiv
ST (State): Kyiv Oblast
C (Country): UA
```

#### 2. **Issuer (Емітент)**
```
Хто видав сертифікат (Certificate Authority)

CN: DigiCert Global Root CA
O: DigiCert Inc
```

#### 3. **Validity Period (Період дії)**
```
Not Before: 2024-01-01 00:00:00 UTC
Not After:  2024-12-31 23:59:59 UTC

Максимальний термін (2024): 398 днів (~13 місяців)
Раніше було: 2-3 роки
```

#### 4. **Public Key (Публічний ключ)**
```
Публічний ключ власника сертифіката

Може бути:
- RSA (2048, 3072, 4096 біт)
- ECDSA (P-256, P-384, P-521)
- EdDSA (Ed25519, Ed448)
```

#### 5. **Signature (Підпис CA)**
```
Цифровий підпис від Certificate Authority

Алгоритм: sha256WithRSAEncryption
Підпис: (бінарні дані)

Це доказ, що CA перевірив і підписав цей сертифікат
```

### Subject Alternative Names (SAN)

**Сучасні сертифікати використовують SAN для підтримки багатьох доменів:**

```
SAN може містити:

DNS Names:
  - example.com
  - www.example.com
  - api.example.com
  - *.example.com (wildcard)

IP Addresses:
  - 192.168.1.1
  - 2001:db8::1

Email Addresses:
  - admin@example.com

URIs:
  - https://example.com
```

**Приклад Wildcard сертифіката:**
```
CN: *.example.com

Покриває:
✅ www.example.com
✅ api.example.com
✅ mail.example.com

НЕ покриває:
❌ example.com (без субдомену)
❌ subdomain.api.example.com (два рівні)
```

### Public Key Infrastructure (PKI)

**PKI** — це ієрархічна система довіри для управління сертифікатами.

```
                    ┌─────────────────────┐
                    │   Root CA           │
                    │  (Довірений корінь)  │
                    └──────────┬──────────┘
                               │
              ┌────────────────┴─────────────────┐
              │                                  │
     ┌────────▼────────┐              ┌─────────▼─────────┐
     │ Intermediate CA 1│              │ Intermediate CA 2 │
     │  (Проміжний CA)  │              │  (Проміжний CA)   │
     └────────┬─────────┘              └─────────┬─────────┘
              │                                  │
     ┌────────┴─────────┐              ┌────────┴─────────┐
     │                  │              │                  │
┌────▼────┐      ┌─────▼─────┐  ┌────▼────┐      ┌─────▼─────┐
│Сертифікат│      │Сертифікат│  │Сертифікат│      │Сертифікат│
│example.com│     │test.com  │  │shop.com │      │api.com   │
└──────────┘      └──────────┘  └─────────┘      └──────────┘
```

### Типи Certificate Authorities:

#### 1. **Root CA (Кореневий CA)**

```
Найвищий рівень довіри

Характеристики:
- Самопідписаний сертифікат
- Попередньо встановлений у браузерах/ОС
- Дуже захищений (часто offline)
- Підписує Intermediate CA

Приклади:
- DigiCert Global Root CA
- Let's Encrypt ISRG Root X1
- GlobalSign Root CA
```

**Як Root CA потрапляє у браузер:**
```
1. CA проходить аудит (WebTrust, ETSI)
2. Подає заявку до Mozilla, Google, Apple, Microsoft
3. Після перевірки → додається до Root Store
4. Браузер/ОС довіряє цьому CA

В Windows:
Win + R → certmgr.msc → Trusted Root Certification Authorities

Кількість довірених Root CA: ~50-150 (залежить від ОС)
```

#### 2. **Intermediate CA (Проміжний CA)**

```
Підписаний Root CA, підписує кінцеві сертифікати

Навіщо:
- Безпека: Root CA offline, Intermediate — online
- Якщо Intermediate скомпрометовано → відкликати тільки його
- Root CA залишається безпечним

Ланцюжок довіри:
End Certificate → Intermediate CA → Root CA
```

#### 3. **End-Entity Certificate (Кінцевий сертифікат)**

```
Сертифікат веб-сайту, який ви бачите у браузері

Підписаний Intermediate CA
Використовується для TLS з'єднань
```

### Certificate Chain (Ланцюжок сертифікатів)

```
Перевірка сертифіката example.com:

┌─────────────────────────────────────────┐
│ 1. End Certificate (example.com)        │ ← Отримано від сервера
│    Issued by: Let's Encrypt R3          │
│    Signature: [підпис]                  │
└───────────────┬─────────────────────────┘
                │
                │ Перевірка підпису
                ↓
┌─────────────────────────────────────────┐
│ 2. Intermediate Certificate (R3)        │ ← Отримано від сервера
│    Issued by: ISRG Root X1              │
│    Signature: [підпис]                  │
└───────────────┬─────────────────────────┘
                │
                │ Перевірка підпису
                ↓
┌─────────────────────────────────────────┐
│ 3. Root Certificate (ISRG Root X1)      │ ← Попередньо встановлений
│    Issued by: ISRG Root X1 (self-signed)│    у браузері
│    Signature: [підпис]                  │
└─────────────────────────────────────────┘
                │
                ↓
            ✅ ДОВІРЕНИЙ!
```

**Процес перевірки:**
```
1. Браузер отримує сертифікат example.com від сервера
2. Перевіряє підпис за допомогою публічного ключа R3 (Intermediate)
3. Перевіряє підпис R3 за допомогою публічного ключа ISRG Root X1
4. ISRG Root X1 є у Root Store браузера → ✅ Довіряємо!
5. Якщо всі підписи валідні → з'єднання безпечне
```

### Типи сертифікатів (за рівнем перевірки):

#### 1. **Domain Validation (DV)**

```
Базова перевірка - тільки володіння доменом

Процес отримання:
1. Запит на сертифікат для example.com
2. CA перевіряє, що ви контролюєте домен:
   - Розміщення файлу на сервері (HTTP)
   - DNS запис (TXT record)
   - Email на admin@example.com
3. Якщо перевірка пройшла → видає сертифікат (автоматично)

Час: Хвилини - години
Вартість: Безкоштовно (Let's Encrypt) або недорого
Відображення у браузері: 🔒 (стандартний замок)

Використання:
✅ Блоги, особисті сайти
✅ Невеликі бізнеси
✅ API endpoints

Не містить інформації про організацію
```

#### 2. **Organization Validation (OV)**

```
Перевірка організації

Процес отримання:
1. Перевірка володіння доменом (як DV)
2. Перевірка існування організації:
   - Перевірка реєстраційних даних
   - Дзвінок у компанію
   - Перевірка документів
3. Видача сертифіката

Час: 1-3 дні
Вартість: $50-$200/рік
Відображення: 🔒 + інформація про організацію (у деталях)

Сертифікат містить:
- Назву організації
- Місцезнаходження
- Країну

Використання:
✅ Корпоративні сайти
✅ E-commerce
✅ B2B сервіси
```

#### 3. **Extended Validation (EV)**

```
Найвища перевірка

Процес отримання:
1. Перевірка володіння доменом
2. Детальна перевірка організації:
   - Юридичне існування (3+ роки)
   - Фізична адреса
   - Телефонний номер
   - Співробітники
   - Перевірка через бази даних Dun & Bradstreet
3. Особистий дзвінок керівництву
4. Видача сертифіката

Час: 1-2 тижні
Вартість: $100-$300/рік
Відображення (старі браузери): 🔒 + зелена адресна строка з назвою компанії

Примітка: Сучасні браузери (2019+) прибрали зелену строку,
тепер виглядає як звичайний DV :(

Використання:
✅ Банки
✅ Фінансові організації
✅ Великі e-commerce
```

**Порівняння:**

| Тип | Перевірка | Час | Вартість | Візуальна відмінність (2024) |
|-----|-----------|-----|----------|------------------------------|
| DV  | Домен     | Хвилини | $0-$50 | 🔒 |
| OV  | Організація | 1-3 дні | $50-$200 | 🔒 (інфо в деталях) |
| EV  | Розширена | 1-2 тижні | $100-$300 | 🔒 (інфо в деталях) |

### Спеціальні типи сертифікатів:

#### 1. **Wildcard Certificate**

```
Покриває всі субдомени одного рівня

*.example.com

Покриває:
✅ www.example.com
✅ api.example.com
✅ mail.example.com
✅ anything.example.com

НЕ покриває:
❌ example.com (основний домен без субдомену)
❌ sub.api.example.com (два рівні)

Вартість: Зазвичай дорожче за звичайний DV
```

#### 2. **Multi-Domain / SAN Certificate**

```
Один сертифікат для багатьох доменів

SAN містить:
- example.com
- example.net
- example.org
- www.example.com
- shop.example.com

Зазвичай: до 100 доменів в одному сертифікаті

Використання:
✅ Кілька доменів однієї компанії
✅ Економія (один сертифікат замість багатьох)
```

#### 3. **Self-Signed Certificate (Самопідписаний)**

```
Створений і підписаний власноруч (без CA)

Генерація:
openssl req -x509 -newkey rsa:2048 -keyout key.pem -out cert.pem -days 365

Проблема:
❌ Браузери НЕ довіряють
⚠️ Попередження: "Ваше з'єднання не є приватним"

Використання:
✅ Локальна розробка (localhost)
✅ Внутрішні мережі (intranet)
✅ Тестування

❌ НІКОЛИ для публічних сайтів!
```

### Certificate Authorities (приклади):

#### Безкоштовні:

**Let's Encrypt**
```
Найпопулярніший безкоштовний CA

Характеристики:
- Автоматизований (ACME протокол)
- DV сертифікати
- Wildcard підтримка
- Термін: 90 днів (автопродовження)

Використовується:
- 300+ мільйонів сайтів
- Підтримка у всіх хостингах

Автоматизація:
certbot --nginx -d example.com
```

**ZeroSSL**

### Certificate Revocation (Відкликання сертифікатів)

Іноді сертифікат потрібно відкликати до закінчення терміну дії:

**Причини відкликання:**
```
1. Приватний ключ скомпрометовано
2. Організація змінилась
3. Помилка у сертифікаті
4. CA скомпрометований
```

#### CRL (Certificate Revocation List)

```
Список відкликаних сертифікатів

Як працює:
1. CA публікує CRL (файл зі списком відкликаних серійних номерів)
2. Браузер завантажує CRL
3. Перевіряє, чи є сертифікат у списку

Проблеми:
❌ Великий розмір (мегабайти)
❌ Повільно
❌ Може бути застарілим

Статус: Застарілий метод
```

#### OCSP (Online Certificate Status Protocol)

```
Перевірка статусу сертифіката онлайн

Як працює:
1. Браузер відправляє запит до OCSP responder CA
   "Чи валідний сертифікат з серійним номером 123456?"
2. OCSP responder відповідає:
   - Good (валідний)
   - Revoked (відкликаний)
   - Unknown (невідомий)

Переваги:
✅ Швидко
✅ Актуально

Проблеми:
❌ Проблема приватності (CA бачить, які сайти ви відвідуєте)
❌ Якщо OCSP сервер недоступний → часто ігнорується (soft-fail)
```

#### OCSP Stapling

```
Покращена версія OCSP

Як працює:
1. СЕРВЕР отримує OCSP відповідь від CA (кожні кілька годин)
2. При TLS handshake сервер надсилає OCSP відповідь разом із сертифікатом
3. Браузер перевіряє підпис CA на OCSP відповіді

Переваги:
✅ Швидко (немає додаткового запиту)
✅ Приватність (CA не знає, що ви відвідуєте сайт)
✅ Надійно (навіть якщо OCSP сервер недоступний)

Статус: ✅ Рекомендовано (2024)

Nginx конфіг:
ssl_stapling on;
ssl_stapling_verify on;
ssl_trusted_certificate /path/to/chain.pem;
```

### Certificate Transparency (CT)

```
Публічний журнал всіх виданих сертифікатів

Мета: Запобігти видачі підроблених сертифікатів

Як працює:
1. CA видає сертифікат
2. CA додає сертифікат до публічного CT log
3. CT log видає SCT (Signed Certificate Timestamp)
4. Браузер перевіряє SCT

Сертифікат БЕЗ SCT → ⚠️ Браузери можуть не довіряти (Chrome вимагає з 2018)

Перевірити сертифікати домену:
https://crt.sh/?q=example.com

Побачите ВСІ сертифікати, коли-небудь видані для домену!
```

### Certificate Pinning

```
Прив'язка сертифіката або публічного ключа

Мета: Захист від MITM атак (навіть із підробленим CA)

Як працює:
Додаток (мобільний або браузер) "запам'ятовує" публічний ключ сервера

Приклад (HTTP заголовок):
Public-Key-Pins: pin-sha256="base64=="; pin-sha256="base64=="; max-age=5184000

Якщо сервер надає інший публічний ключ → з'єднання відхиляється ❌

Використання:
✅ Мобільні додатки (банки)
✅ Критичні сервіси

Примітка: HPKP (HTTP Public Key Pinning) deprecated у браузерах через ризики
```

## TLS Handshake

**TLS Handshake** — це процес встановлення захищеного з'єднання між клієнтом та сервером. Під час handshake відбувається:
1. Узгодження версії TLS та cipher suite
2. Автентифікація сервера (перевірка сертифіката)
3. Обмін ключами
4. Перевірка цілісності handshake

### Загальна схема:

```
         Клієнт                                    Сервер
            │                                         │
            │────── 1. ClientHello ─────────────────→│
            │                                         │
            │←───── 2. ServerHello ──────────────────│
            │←───── 3. Certificate ───────────────────│
            │←───── 4. ServerKeyExchange (опц.) ─────│
            │←───── 5. ServerHelloDone ──────────────│
            │                                         │
            │────── 6. ClientKeyExchange ───────────→│
            │────── 7. ChangeCipherSpec ────────────→│
            │────── 8. Finished ─────────────────────→│
            │                                         │
            │←───── 9. ChangeCipherSpec ─────────────│
            │←───── 10. Finished ────────────────────│
            │                                         │
            │═══════ Зашифроване з'єднання ══════════│
```

### TLS 1.2 Handshake (детально)

#### Крок 1: ClientHello

**Клієнт відправляє:**
```
ClientHello {
  Protocol Version: TLS 1.2
  Random: [32 байти випадкових даних]
  Session ID: [] (порожній для нового з'єднання)
  Cipher Suites: [список підтримуваних]
    - TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
    - TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
    - TLS_ECDHE_RSA_WITH_CHACHA20_POLY1305_SHA256
    - ...
  Compression Methods: null (стиснення deprecated)
  Extensions:
    - server_name: example.com (SNI)
    - supported_groups: P-256, P-384, X25519
    - signature_algorithms: RSA-PSS-SHA256, ECDSA-SHA256
    - application_layer_protocol_negotiation: h2, http/1.1
}
```

**Пояснення полів:**

- **Random**: 32 байти випадкових даних. Використовується для генерації session keys. Захист від replay attacks.
  
- **Cipher Suites**: Список алгоритмів шифрування, які підтримує клієнт, у порядку пріоритету.

- **SNI (Server Name Indication)**: Доменне ім'я, для якого запитується сертифікат. Дозволяє одному серверу обслуговувати кілька доменів.

**Приклад Random:**
```
Client Random: 507f1f77 bcf86cd7 99439011 75847273 c1121fa0 8e79b9aa 88a49a9c 8e5f8d8f
                └─────┬─────┘
                  Timestamp (4 байти) + 28 байтів випадкових даних
```

#### Крок 2: ServerHello

**Сервер відповідає:**
```
ServerHello {
  Protocol Version: TLS 1.2
  Random: [32 байти випадкових даних сервера]
  Session ID: [32 байти] (для session resumption)
  Cipher Suite: TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256 ← вибрано сервером
  Compression Method: null
  Extensions:
    - renegotiation_info
    - extended_master_secret
    - application_layer_protocol_negotiation: h2
}
```

**Сервер вибирає:**
- Одну версію TLS (найвищу спільну)
- Один cipher suite з списку клієнта
- Генерує свій Random

#### Крок 3: Certificate

**Сервер відправляє свій сертифікат:**
```
Certificate {
  Certificate Chain:
    1. Server Certificate (example.com)
       ├─ Subject: CN=example.com
       ├─ Issuer: CN=Let's Encrypt R3
       ├─ Public Key: RSA 2048 біт або ECDSA P-256
       ├─ Validity: 2024-01-01 to 2024-12-31
       └─ Signature: [підпис від Let's Encrypt R3]
    
    2. Intermediate Certificate (Let's Encrypt R3)
       ├─ Subject: CN=Let's Encrypt R3
       ├─ Issuer: CN=ISRG Root X1
       └─ Signature: [підпис від Root CA]
}
```

**Клієнт перевіряє:**
1. Ланцюжок сертифікатів до довіреного Root CA
2. Підписи кожного сертифіката
3. Термін дії
4. Доменне ім'я (SNI) відповідає CN або SAN
5. Статус відкликання (OCSP/CRL)

#### Крок 4: ServerKeyExchange (опціонально)

**Для ECDHE (Elliptic Curve Diffie-Hellman Ephemeral):**

```
ServerKeyExchange {
  Curve Type: named_curve
  Named Curve: x25519 (або P-256, P-384)
  Public Key: [32 байти для X25519]
  Signature Algorithm: rsa_pss_rsae_sha256
  Signature: [підпис ключа приватним ключем сервера]
}
```

**Навіщо:** Для алгоритмів з Perfect Forward Secrecy (ECDHE, DHE) потрібен додатковий обмін ключами.

**Perfect Forward Secrecy (PFS):**
```
Без PFS (RSA key exchange):
Якщо приватний ключ сервера скомпрометовано → можна розшифрувати ВСІ попередні сесії ❌

З PFS (ECDHE):
Кожна сесія має унікальні ephemeral (тимчасові) ключі
Якщо приватний ключ скомпрометовано → попередні сесії залишаються безпечними ✅
```

#### Крок 5: ServerHelloDone

```
ServerHelloDone {}
```

Просто повідомляє клієнту: "Я закінчив свою частину handshake".

#### Крок 6: ClientKeyExchange

**Клієнт генерує та відправляє свою частину ключа:**

**Для ECDHE:**
```
ClientKeyExchange {
  Public Key: [32 байти публічного ключа клієнта]
}
```

**Тепер обидві сторони можуть обчислити спільний секрет:**

```
Клієнт:
Приватний ключ клієнта + Публічний ключ сервера → Shared Secret

Сервер:
Приватний ключ сервера + Публічний ключ клієнта → Shared Secret

Обидва отримали ОДНАКОВИЙ Shared Secret! ✅
```

**З Shared Secret генеруються session keys:**

```
Pre-Master Secret = Shared Secret

Master Secret = PRF(Pre-Master Secret, "master secret", ClientRandom + ServerRandom)

З Master Secret виводяться:
├─ Client Write Key (клієнт шифрує)
├─ Server Write Key (сервер шифрує)
├─ Client Write IV (initialization vector)
├─ Server Write IV
├─ Client Write MAC Key (для HMAC)
└─ Server Write MAC Key
```

#### Крок 7-8: ChangeCipherSpec + Finished (Клієнт)

```
ChangeCipherSpec {
  (Повідомляє: "Далі все буде зашифровано")
}

Finished {
  verify_data = PRF(Master Secret, "client finished", Hash(всі handshake повідомлення))
}
```

**Finished message** — перше зашифроване повідомлення, яке підтверджує:
- Всі handshake повідомлення отримано без змін
- Обидві сторони мають однакові ключі

#### Крок 9-10: ChangeCipherSpec + Finished (Сервер)

Сервер робить те саме:

```
ChangeCipherSpec {}

Finished {
  verify_data = PRF(Master Secret, "server finished", Hash(всі handshake повідомлення))
}
```

Клієнт перевіряє Finished від сервера. Якщо все OK → handshake завершено! ✅

#### Крок 11: Зашифроване з'єднання

```
HTTP Request (зашифроване AES-128-GCM):
GET /api/data HTTP/1.1
Host: example.com
...
```

### TLS 1.3 Handshake

**Головна відмінність:** Швидше! 1-RTT (Round Trip Time) замість 2-RTT.

```
         Клієнт                                    Сервер
            │                                         │
            │────── ClientHello + KeyShare ─────────→│
            │                                         │
            │←───── ServerHello + KeyShare ──────────│
            │←───── {EncryptedExtensions} ───────────│
            │←───── {Certificate} ────────────────────│
            │←───── {CertificateVerify} ──────────────│
            │←───── {Finished} ───────────────────────│
            │                                         │
            │────── {Finished} ───────────────────────→│
            │                                         │
            │═══════ Зашифроване з'єднання ══════════│
            
{ } = зашифровано
```

#### Ключові покращення TLS 1.3:

**1. 1-RTT замість 2-RTT:**

```
TLS 1.2: 2 round trips = ~200ms (при 100ms latency)
TLS 1.3: 1 round trip = ~100ms (при 100ms latency)

Швидше у 2 рази! ⚡
```

**2. Клієнт відправляє KeyShare відразу:**

```http
ClientHello {
  Protocol Version: TLS 1.3
  Random: [32 байти]
  Cipher Suites: [TLS 1.3 only]
    - TLS_AES_128_GCM_SHA256
    - TLS_AES_256_GCM_SHA384
    - TLS_CHACHA20_POLY1305_SHA256
  Key Share:
    - Group: x25519
    - Public Key: [32 байти] ← Клієнт вже відправляє свій публічний ключ!
  Supported Groups: x25519, P-256, P-384
  Signature Algorithms: rsa_pss_rsae_sha256, ecdsa_secp256r1_sha256
}
```

Клієнт "вгадує", який алгоритм обміну ключами вибере сервер, і відправляє KeyShare відразу.

**3. Handshake зашифрований:**

У TLS 1.3 майже весь handshake зашифрований (Certificate, CertificateVerify, Finished).

```
TLS 1.2:
ClientHello → ❌ незашифровано
ServerHello, Certificate → ❌ незашифровано

Хакер може бачити:
- Який сертифікат використовується
- Cipher suites
- Extensions

TLS 1.3:
ClientHello → ❌ незашифровано (неможливо шифрувати)
ServerHello → ❌ незашифровано
Certificate → ✅ зашифровано!
CertificateVerify → ✅ зашифровано!

Хакер бачить менше інформації!
```

**4. Видалено застарілі алгоритми:**

```
TLS 1.2 дозволяє:
- RSA key exchange (без PFS) ❌
- CBC mode ciphers (вразливості) ❌
- SHA-1, MD5 (слабкі хеші) ❌
- Compression (CRIME атака) ❌
- Renegotiation (складно, вразливості) ❌

TLS 1.3 тільки:
- ECDHE або DHE (PFS обов'язково) ✅
- AEAD ciphers (AES-GCM, ChaCha20-Poly1305) ✅
- SHA-256 або краще ✅
- Без compression ✅
- Без renegotiation ✅

Простіше = безпечніше!
```

### TLS 1.3 з 0-RTT (Zero Round Trip Time)

**Для повторних з'єднань:**

```
         Клієнт                                    Сервер
            │                                         │
            │────── ClientHello + 0-RTT Data ────────→│
            │       {HTTP GET /api/data}              │
            │                                         │
            │←───── ServerHello ─────────────────────│
            │←───── {EncryptedExtensions} ───────────│
            │←───── {Finished} ───────────────────────│
            │←───── {HTTP Response} ──────────────────│
            │                                         │
            │────── {Finished} ───────────────────────→│
```

**Перевага:** HTTP запит надсилається ОДНОЧАСНО з ClientHello → 0 RTT! Миттєво! ⚡⚡⚡

**Застереження 0-RTT:**
```
Проблема: 0-RTT дані НЕ захищені від replay attacks!

Атака:
1. Хакер перехоплює ClientHello + 0-RTT Data з HTTP POST запитом
2. Хакер відправляє той самий пакет ще раз (replay)
3. Сервер може обробити запит двічі!

Приклад атаки:
0-RTT: POST /transfer?amount=1000
Replay → гроші переведено двічі! ❌

Безпечно для 0-RTT:
✅ GET запити (ідемпотентні)
✅ Дані, які можна повторювати

НЕ безпечно:
❌ POST, PUT, DELETE
❌ Транзакції
❌ Зміна стану

Статус: 0-RTT використовується обережно, часто вимкнено для POST/PUT/DELETE
```

### Session Resumption

**Мета:** Прискорити повторні з'єднання (не потрібен повний handshake).

#### Session ID (TLS 1.2)

```
Перше з'єднання:
ClientHello { Session ID: [] } ← порожній

ServerHello { Session ID: abc123 } ← сервер генерує ID
→ Сервер зберігає session keys під цим ID

Повторне з'єднання:
ClientHello { Session ID: abc123 } ← клієнт відправляє збережений ID

ServerHello { Session ID: abc123 } ← сервер знаходить session keys
ChangeCipherSpec + Finished ← одразу використовують збережені ключі

Handshake скорочено! ⚡
```

**Проблема:** Сервер має зберігати session state (проблема для load balancing).

#### Session Tickets (TLS 1.2/1.3)

```
Перше з'єднання:
После handshake сервер відправляє:

NewSessionTicket {
  Ticket: [зашифровані session keys + metadata]
  (зашифровано секретним ключем сервера)
}

Клієнт зберігає ticket

Повторне з'єднання:
ClientHello {
  Extension: session_ticket
  Ticket: [збережений ticket]
}

Сервер розшифровує ticket → отримує session keys → abbreviated handshake

Перевага: Сервер НЕ зберігає state! (stateless resumption)
```

#### PSK (Pre-Shared Key) у TLS 1.3

```
TLS 1.3 використовує PSK для resumption:

После першого з'єднання сервер відправляє:
NewSessionTicket {
  ticket_lifetime: 86400 (24 години)
  ticket_age_add: [random]
  ticket_nonce: [nonce]
  ticket: [encrypted state]
  extensions: early_data (якщо підтримується 0-RTT)
}

Повторне з'єднання:
ClientHello {
  Extension: pre_shared_key
  Identities: [ticket]
  Binders: [HMAC proof of PSK]
}

ServerHello {
  Extension: pre_shared_key ← підтверджує PSK
}

Abbreviated handshake + можливість 0-RTT
```

## Cipher Suites

**Cipher Suite** — це набір криптографічних алгоритмів, який визначає, як буде захищене TLS з'єднання.

### Анатомія Cipher Suite

**Формат назви (TLS 1.2):**
```
TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
│   │     │    │    │   │   │   │
│   │     │    │    │   │   │   └─ Hash для PRF (Pseudo-Random Function)
│   │     │    │    │   │   └───── Hash для MAC (Message Authentication Code)
│   │     │    │    │   └────────── Режим роботи (GCM, CBC)
│   │     │    │    └────────────── Розмір ключа (128, 256 біт)
│   │     │    └─────────────────── Алгоритм шифрування (AES, ChaCha20)
│   │     └──────────────────────── WITH (сполучник)
│   └────────────────────────────── Алгоритм підпису сертифіката (RSA, ECDSA)
└──────────────────────────────────── Алгоритм обміну ключами (ECDHE, DHE, RSA)
```

**Формат назви (TLS 1.3):**
```
TLS_AES_128_GCM_SHA256
│   │   │   │   │
│   │   │   │   └─ Hash для HKDF (key derivation)
│   │   │   └───── Режим роботи (завжди AEAD)
│   │   └────────── Розмір ключа
│   └────────────── Алгоритм шифрування
└──────────────────── Протокол

Примітка: У TLS 1.3 немає key exchange та signature algorithm у назві,
бо вони узгоджуються окремо через extensions
```

### Компоненти Cipher Suite:

#### 1. Key Exchange (Обмін ключами)

**RSA** (застарілий у TLS 1.3):
```
Клієнт генерує pre-master secret
Шифрує його публічним ключем з сертифіката сервера
Відправляє серверу

Недолік: Немає Perfect Forward Secrecy
Якщо приватний ключ сервера скомпрометовано → можна розшифрувати старі сесії
```

**DHE (Diffie-Hellman Ephemeral)**:
```
Алгоритм Diffie-Hellman з тимчасовими ключами

Переваги:
✅ Perfect Forward Secrecy
✅ Кожна сесія має унікальні ключі

Недоліки:
❌ Повільніше за RSA
❌ Великі ключі (2048+ біт)
```

**ECDHE (Elliptic Curve Diffie-Hellman Ephemeral)** ✅ Рекомендовано:
```
Diffie-Hellman на еліптичних кривих

Переваги:
✅ Perfect Forward Secrecy
✅ Швидше за DHE
✅ Менші ключі (256 біт ECDHE ≈ 3072 біт DHE)

Криві:
- X25519 (найпопулярніша, найшвидша)
- P-256 (secp256r1)
- P-384 (secp384r1)
- P-521 (secp521r1)

Статус: ✅ Обов'язковий у TLS 1.3
```

#### 2. Authentication (Автентифікація)

**RSA**:
```
Підпис сертифіката створений за допомогою RSA

Розміри: 2048, 3072, 4096 біт
Швидкість: Повільно
Статус: ✅ Досі використовується
```

**ECDSA** ✅ Рекомендовано:
```
Підпис на еліптичних кривих

Розміри: P-256, P-384, P-521
Швидкість: Швидше за RSA
Розмір підпису: Менший
Статус: ✅ Рекомендовано
```

**EdDSA** ✅ Найкраще:
```
Ed25519, Ed448

Найшвидший
Найбезпечніший
Статус: ✅ Підтримується у TLS 1.3
```

#### 3. Encryption (Шифрування даних)

**AES (Advanced Encryption Standard)** ✅:
```
Найпопулярніший алгоритм

Розміри ключів:
- AES-128: 128 біт (достатньо для більшості)
- AES-256: 256 біт (максимальна безпека)

Швидкість: Дуже швидко (особливо з AES-NI інструкціями CPU)
Безпека: Відмінна
Статус: ✅ Рекомендовано
```

**ChaCha20** ✅:
```
Сучасний потоковий шифр від Daniel Bernstein

Розмір ключа: 256 біт
Швидкість: Дуже швидко на мобільних пристроях (без AES-NI)
Безпека: Відмінна
Статус: ✅ Рекомендовано (особливо для мобільних)

Переваги на мобільних:
- ARM процесори без AES-NI → ChaCha20 швидше за AES
- Intel/AMD з AES-NI → AES швидше
```

**3DES (Triple DES)** ⚠️ Застарілий:
```
Потрійне застосування DES

Розмір ключа: 168 біт (ефективно ~112 біт)
Швидкість: Повільно
Безпека: Слабка за сучасними стандартами
Статус: ⚠️ Deprecated, НЕ використовувати!
```

**RC4** ❌ Заборонений:
```
Застарілий потоковий шифр

Статус: ❌ Зламаний, ЗАБОРОНЕНИЙ у всіх версіях TLS!
```

#### 4. Mode of Operation (Режим роботи)

**GCM (Galois/Counter Mode)** ✅ AEAD:
```
Authenticated Encryption with Associated Data

Переваги:
✅ Вбудована автентифікація (не потрібен окремий MAC)
✅ Паралелізується (швидко)
✅ Захист від padding oracle атак

Статус: ✅ Рекомендовано
```

**CBC (Cipher Block Chaining)** ⚠️:
```
Класичний режим блокового шифрування

Проблеми:
❌ Вразливий до padding oracle атак (BEAST, Lucky 13)
❌ Потребує окремого MAC (HMAC)
❌ Не паралелізується

Статус: ⚠️ Застарілий, уникати
```

**CCM (Counter with CBC-MAC)** ✅ AEAD:
```
Альтернатива GCM

Переваги:
✅ AEAD
✅ Менше пам'яті

Недоліки:
❌ Не паралелізується
❌ Повільніше за GCM

Статус: ✅ OK, але GCM кращий
```

**Poly1305 (з ChaCha20)** ✅ AEAD:
```
ChaCha20-Poly1305

Poly1305 - MAC алгоритм для автентифікації

Переваги:
✅ AEAD
✅ Швидко на мобільних
✅ Простіше для безпечної реалізації

Статус: ✅ Рекомендовано
```

#### 5. MAC (Message Authentication Code)

**Для режимів без AEAD (CBC, старі):**

**SHA256**:
```
Використовується у HMAC-SHA256

Безпека: ✅ Хороша
Статус: ✅ OK для TLS 1.2 з CBC (хоча GCM краще)
```

**SHA384**:
```
Використовується з AES-256

Безпека: ✅ Відмінна
```

**SHA1** ⚠️ Застарілий:
```
Статус: ⚠️ Вразливий, не рекомендується
```

**MD5** ❌ Заборонений:
```
Статус: ❌ Зламаний, ЗАБОРОНЕНИЙ!
```

### Популярні Cipher Suites:

#### TLS 1.3 (тільки AEAD):

```
✅ TLS_AES_128_GCM_SHA256
   - Найпопулярніший
   - AES-128 з GCM
   - Швидкий на системах з AES-NI
   - Достатня безпека для більшості

✅ TLS_AES_256_GCM_SHA384
   - Максимальна безпека
   - AES-256 з GCM
   - Трохи повільніше за AES-128

✅ TLS_CHACHA20_POLY1305_SHA256
   - Кращий для мобільних пристроїв
   - Швидкий без AES-NI
   - Простіше для безпечної реалізації

❌ TLS_AES_128_CCM_SHA256
   - Рідко використовується
   - Повільніше за GCM

❌ TLS_AES_128_CCM_8_SHA256
   - Скорочений MAC (8 байт замість 16)
   - Для IoT з обмеженою пам'яттю
```

**Рекомендований порядок (TLS 1.3):**
```
1. TLS_AES_256_GCM_SHA384 (найбезпечніший)
2. TLS_CHACHA20_POLY1305_SHA256 (для мобільних)
3. TLS_AES_128_GCM_SHA256 (найшвидший з AES-NI)
```

#### TLS 1.2:

**Рекомендовані (з PFS):**

```
✅ TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
   - Найпопулярніший для TLS 1.2
   - ECDHE для PFS
   - RSA сертифікат
   - AES-128-GCM

✅ TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
   - Максимальна безпека
   - AES-256

✅ TLS_ECDHE_RSA_WITH_CHACHA20_POLY1305_SHA256
   - Для мобільних
   - ChaCha20-Poly1305

✅ TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256
   - ECDSA сертифікат (швидше за RSA)
   - AES-128-GCM

✅ TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384
   - ECDSA + максимальна безпека

✅ TLS_ECDHE_ECDSA_WITH_CHACHA20_POLY1305_SHA256
   - ECDSA + ChaCha20 (найшвидше на мобільних)
```

**Застарілі (але все ще широко використовувані):**

```
⚠️ TLS_ECDHE_RSA_WITH_AES_128_CBC_SHA256
   - CBC режим (padding oracle вразливості)
   - Краще замінити на GCM

⚠️ TLS_ECDHE_RSA_WITH_AES_256_CBC_SHA384
   - CBC режим
   - Краще замінити на GCM

⚠️ TLS_DHE_RSA_WITH_AES_128_GCM_SHA256
   - DHE замість ECDHE (повільніше)
   - OK, але ECDHE краще
```

**НЕ рекомендовані (без PFS):**

```
❌ TLS_RSA_WITH_AES_128_GCM_SHA256
   - RSA key exchange
   - БЕЗ Perfect Forward Secrecy
   - Уникати!

❌ TLS_RSA_WITH_AES_256_GCM_SHA384
   - Те саме

❌ TLS_RSA_WITH_AES_128_CBC_SHA256
   - RSA + CBC
   - Подвійна проблема!

❌ TLS_RSA_WITH_3DES_EDE_CBC_SHA
   - 3DES застарілий
   - CBC вразливий
   - НЕ використовувати!
```

**Заборонені:**

```
❌ TLS_RSA_WITH_RC4_128_SHA
   - RC4 зламаний!
   - ЗАБОРОНЕНО!

❌ TLS_RSA_WITH_RC4_128_MD5
   - RC4 + MD5
   - Повний провал!

❌ TLS_NULL_WITH_NULL_NULL
   - Без шифрування взагалі!
   - Тільки для тестування
```

### Як вибрати Cipher Suite?

**Пріоритети:**

```
1. Perfect Forward Secrecy (PFS)
   → Використовувати ECDHE або DHE
   → Уникати RSA key exchange

2. AEAD шифрування
   → GCM, ChaCha20-Poly1305
   → Уникати CBC

3. Сучасні алгоритми
   → AES-128/256, ChaCha20
   → Уникати 3DES, RC4

4. Сильні хеші
   → SHA-256, SHA-384
   → Уникати SHA-1, MD5
```

**Для веб-серверів (2024):**

```nginx
# Nginx конфігурація
ssl_protocols TLSv1.2 TLSv1.3;

ssl_ciphers 'ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256';

ssl_prefer_server_ciphers off; # Дозволити клієнту вибирати (TLS 1.3 стиль)
```

```apache
# Apache конфігурація
SSLProtocol -all +TLSv1.2 +TLSv1.3

SSLCipherSuite ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256

SSLHonorCipherOrder off
```

### Cipher Suite вибір під час handshake:

```
TLS 1.2:
1. Клієнт відправляє список cipher suites (у порядку пріоритету)
2. Сервер вибирає ПЕРШИЙ з списку, який він підтримує
3. Або: сервер вибирає зі свого списку пріоритетів (ssl_prefer_server_ciphers on)

TLS 1.3:
1. Клієнт відправляє список
2. Сервер вибирає зі списку клієнта
3. Зазвичай клієнт має пріоритет (ssl_prefer_server_ciphers off)
```

**Приклад:**

```
ClientHello:
  Cipher Suites:
    1. TLS_AES_256_GCM_SHA384
    2. TLS_CHACHA20_POLY1305_SHA256
    3. TLS_AES_128_GCM_SHA256
    4. TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256 (TLS 1.2)

ServerHello:
  Cipher Suite: TLS_AES_256_GCM_SHA384 ← вибрано сервером

Обидві сторони тепер використовують AES-256-GCM!
```

## Версії протоколу

### Хронологія версій:

```
1994 - SSL 1.0 (ніколи не випущений)
1995 - SSL 2.0 ⚠️ ЗАБОРОНЕНИЙ
1996 - SSL 3.0 ⚠️ ЗАБОРОНЕНИЙ
1999 - TLS 1.0 ⚠️ ЗАБОРОНЕНИЙ з 2020
2006 - TLS 1.1 ⚠️ ЗАБОРОНЕНИЙ з 2020
2008 - TLS 1.2 ✅ OK (але TLS 1.3 краще)
2018 - TLS 1.3 ✅ РЕКОМЕНДОВАНО
```

### SSL 2.0 (1995) ❌

**Проблеми:**
```
❌ Слабкі cipher suites (MD5, 40-bit encryption)
❌ Вразливості в дизайні
❌ Відсутність PFS
❌ Downgrade атаки

Статус: ЗАБОРОНЕНИЙ у всіх браузерах з 2011
```

### SSL 3.0 (1996) ❌

**Проблеми:**
```
❌ POODLE атака (2014)
❌ Слабкі cipher suites
❌ Вразливості в padding

POODLE (Padding Oracle On Downgraded Legacy Encryption):
Атака, яка дозволяє розшифрувати CBC-шифровані дані

Статус: ЗАБОРОНЕНИЙ з 2015
```

### TLS 1.0 (1999) ⚠️

**Покращення порівняно з SSL 3.0:**
```
✅ Виправлено деякі вразливості SSL 3.0
✅ HMAC замість MAC
✅ Більш надійна криптографія

Проблеми:
❌ BEAST атака (2011)
❌ Підтримує слабкі cipher suites
❌ CBC вразливості

BEAST (Browser Exploit Against SSL/TLS):
Атака на CBC режим у TLS 1.0

Статус: ЗАБОРОНЕНИЙ з 2020 (PCI DSS, браузери)
```

### TLS 1.1 (2006) ⚠️

**Покращення:**
```
✅ Захист від BEAST (explicit IV)
✅ Захист від CBC атак

Проблеми:
❌ Все ще підтримує слабкі cipher suites
❌ Немає AEAD
❌ Немає обов'язкового PFS

Статус: ЗАБОРОНЕНИЙ з 2020 (разом з TLS 1.0)
```

### TLS 1.2 (2008) ✅

**Основні покращення:**
```
✅ AEAD cipher suites (GCM, CCM)
✅ SHA-256 та кращі хеші
✅ Можливість вказати hash для підпису
✅ Authenticated Encryption
✅ Виправлено багато вразливостей

Підтримувані features:
- AES-GCM ✅
- ChaCha20-Poly1305 ✅
- ECDHE для PFS ✅
- ECDSA сертифікати ✅
- Extensions (SNI, ALPN, OCSP stapling) ✅

Статус: ✅ OK для використання (2024)
Але TLS 1.3 краще!
```

**Конфігурація TLS 1.2:**
```nginx
# Nginx - тільки безпечні cipher suites
ssl_protocols TLSv1.2 TLSv1.3;
ssl_ciphers 'ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256';
ssl_prefer_server_ciphers off;
```

### TLS 1.3 (2018) ✅ РЕКОМЕНДОВАНО

**Революційні зміни:**

**1. Швидше:**
```
TLS 1.2: 2-RTT handshake = ~200ms
TLS 1.3: 1-RTT handshake = ~100ms
TLS 1.3 resumption: 0-RTT = ~0ms

Швидше у 2 рази! ⚡
```

**2. Простіше і безпечніше:**
```
Видалено:
❌ RSA key exchange (без PFS)
❌ DHE (залишився ECDHE)
❌ CBC режим
❌ SHA-1, MD5
❌ RC4, 3DES
❌ Compression
❌ Renegotiation
❌ Custom DHE groups
❌ Static RSA

Тільки:
✅ ECDHE або DHE (PFS обов'язково)
✅ AEAD тільки (GCM, CCM, Poly1305)
✅ SHA-256+
✅ AES, ChaCha20 тільки
✅ Без compression
✅ Без renegotiation

Результат: Менше вибору = менше помилок = безпечніше!
```

**3. Зашифрований handshake:**
```
TLS 1.2:
❌ Certificate відправляється незашифрованим
❌ Хакер бачить, який сайт ви відвідуєте

TLS 1.3:
✅ Certificate зашифровано
✅ Більше приватності
```

**4. Тільки 5 cipher suites:**
```
TLS 1.3 cipher suites (всього 5!):
1. TLS_AES_256_GCM_SHA384
2. TLS_CHACHA20_POLY1305_SHA256
3. TLS_AES_128_GCM_SHA256
4. TLS_AES_128_CCM_SHA256
5. TLS_AES_128_CCM_8_SHA256

Проти 37+ у TLS 1.2!

Простіше конфігурувати ✅
Менше помилок ✅
```

**5. Відокремлені key exchange та authentication:**
```
TLS 1.2:
Cipher suite містить: key exchange + auth + encryption + MAC
TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
    └─┬──┘ └─┬┘      └─────┬─────┘
   Key Exch Auth      Encryption+MAC

TLS 1.3:
Cipher suite містить тільки: encryption + hash
TLS_AES_128_GCM_SHA256
    └─────┬─────┘
   Encryption+Hash

Key exchange та authentication узгоджуються через extensions:
- supported_groups: x25519, P-256, P-384
- signature_algorithms: rsa_pss_rsae_sha256, ecdsa_secp256r1_sha256

Переваги:
✅ Більш гнучко
✅ Можна змінити key exchange без зміни cipher suite
```

**Конфігурація TLS 1.3:**
```nginx
# Nginx
ssl_protocols TLSv1.2 TLSv1.3;

# Для TLS 1.3 cipher suites не потрібно вказувати (тільки 5 варіантів)
# Але можна:
ssl_conf_command Ciphersuites TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256:TLS_AES_128_GCM_SHA256;
```

### Downgrade атаки та захист:

**Проблема:**
```
Хакер перехоплює ClientHello
Змінює список підтримуваних версій:
TLS 1.3, TLS 1.2 → SSL 3.0

Сервер думає, що клієнт підтримує тільки SSL 3.0
→ Використовує вразливу версію!
```

**Захист (TLS 1.3):**
```
ServerHello містить спеціальне значення у Random полі

Якщо сервер "downgrade" до TLS 1.2:
Server Random закінчується на: 44 4F 57 4E 47 52 44 01
                                 D  O  W  N  G  R  D

Якщо до TLS 1.1 або менше:
Server Random закінчується на: 44 4F 57 4E 47 52 44 00
                                 D  O  W  N  G  R  D

Клієнт перевіряє це значення:
- Якщо клієнт підтримує TLS 1.3, але бачить DOWNGRD → розриває з'єднання
- Захист від downgrade атак ✅
```

## Безпека та вразливості

### Історичні атаки на TLS/SSL:

#### 1. BEAST (2011)

**Browser Exploit Against SSL/TLS**

```
Ціль: TLS 1.0 з CBC cipher suites

Як працює:
1. Хакер контролює частину plaintext (через JavaScript на сторінці)
2. Використовує predictable IV у CBC
3. Відгадує байти cookies по одному

Атака:
- Man-in-the-middle позиція
- Ін'єкція JavaScript
- Крадіжка session cookies

Захист:
✅ TLS 1.1+ (explicit IV)
✅ RC4 cipher (на той час, зараз RC4 вразливий)
✅ 1/n-1 record splitting

Статус: Виправлено у TLS 1.1+
```

#### 2. CRIME (2012)

**Compression Ratio Info-leak Made Easy**

```
Ціль: TLS compression

Як працює:
1. TLS compression стискає повторювані дані
2. Якщо cookie містить певні байти → розмір зменшується
3. Хакер ін'єктує різні значення
4. Вимірює розмір зашифрованих даних
5. Знаходить збіг → витягує cookies

Приклад:
Cookie: secret=abc123
Спроба 1: "secret=a" → розмір X
Спроба 2: "secret=b" → розмір X
Спроба 3: "secret=c" → розмір X-1 ← збіг! Перший символ 'a'

Захист:
✅ Вимкнути TLS compression

Статус: TLS compression deprecated
```

#### 3. BREACH (2013)

**Browser Reconnaissance and Exfiltration via Adaptive Compression of Hypertext**

```
Ціль: HTTP compression (не TLS!)

Подібна до CRIME, але атакує HTTP compression (gzip)

Як працює:
1. CSRF атака → змушує жертву робити запити
2. Вимірює розмір відповідей
3. Витягує CSRF tokens, secrets з HTML

Захист:
⚠️ Складно! HTTP compression корисний
✅ Randomize secrets (додати випадкові дані)
✅ Disable compression для чутливих даних
✅ CSRF tokens не в тілі відповіді
✅ Same-site cookies

Статус: Все ще актуальна, складна для експлуатації
```

#### 4. Lucky 13 (2013)

```
Ціль: CBC padding в TLS

Як працює:
Timing attack на MAC verification у CBC режимі

Різниця в часі обробки:
- Правильний padding → перевіряється MAC → довше
- Неправильний padding → помилка → швидше

Хакер вимірює час відповідей → витягує plaintext

Захист:
✅ Constant-time implementation
✅ Уникати CBC, використовувати GCM

Статус: Виправлено у реалізаціях, але CBC все ще вразливий
```

#### 5. POODLE (2014)

**Padding Oracle On Downgraded Legacy Encryption**

```
Ціль: SSL 3.0

Як працює:
1. Хакер примушує downgrade до SSL 3.0 (MITM)
2. SSL 3.0 CBC padding не автентифікований
3. Padding oracle attack → витягує cookies

Атака:
256 запитів → 1 байт cookies

Захист:
✅ Вимкнути SSL 3.0
✅ TLS_FALLBACK_SCSV (захист від downgrade)

Статус: SSL 3.0 заборонений з 2015
```

**POODLE 2.0 (TLS):**
```
Виявлено, що деякі TLS 1.0-1.2 реалізації також вразливі

Захист:
✅ Уникати CBC
✅ Використовувати GCM
```

#### 6. Heartbleed (2014)

**OpenSSL вразливість (CVE-2014-0160)**

```
Ціль: OpenSSL 1.0.1 - 1.0.1f

Як працює:
1. TLS Heartbeat extension (keep-alive)
2. Клієнт: "Відправ мені назад X байтів"
3. Вразливість: не перевіряється розмір
4. Клієнт може попросити більше → читає пам'ять сервера!

Можна витягти:
❌ Приватні ключі сервера!
❌ User credentials
❌ Session tokens
❌ Інші дані з пам'яті

Експлойт:
heartbleed-poc exploit → читає 64KB пам'яті за запит

Захист:
✅ Оновити OpenSSL до 1.0.1g+
✅ Перевидати сертифікати (приватні ключі скомпрометовано!)
✅ Змінити паролі користувачів

Статус: Виправлено, але один з найсерйозніших багів в історії
```

#### 7. FREAK (2015)

**Factoring RSA Export Keys**

```
Ціль: Export-grade cipher suites (512-bit RSA)

Історія:
У 1990-х США обмежувала експорт сильного шифрування
→ Слабкі "export" cipher suites (RSA 512-bit)

Як працює:
1. MITM хакер примушує клієнта використати export cipher
2. 512-bit RSA можна зламати за ~7 годин (Amazon EC2 $100)
3. Розшифровує TLS трафік

Захист:
✅ Вимкнути export cipher suites
✅ Оновити OpenSSL/браузери

Статус: Export ciphers заборонені
```

#### 8. Logjam (2015)

```
Ціль: DHE export cipher suites (512-bit DH)

Подібна до FREAK, але для Diffie-Hellman

512-bit DH можна зламати

Захист:
✅ Вимкнути export ciphers
✅ Використовувати DH 2048+ біт
✅ Або ECDHE замість DHE

Статус: Виправлено
```

#### 9. DROWN (2016)

**Decrypting RSA with Obsolete and Weakened eNcryption**

```
Ціль: Сервери, які підтримують SSLv2

Як працює:
1. Сервер підтримує TLS 1.2 ТА SSLv2 (навіть на іншому порті)
2. Хакер перехоплює TLS 1.2 трафік (RSA key exchange)
3. Використовує SSLv2 для padding oracle attack
4. Витягує RSA приватний ключ
5. Розшифровує TLS 1.2 сесії!

Захист:
✅ Вимкнути SSLv2 ВСЮДИ
✅ Використовувати ECDHE (PFS)

Статус: SSLv2 заборонений
```

#### 10. Sweet32 (2016)

```
Ціль: 64-bit block ciphers (3DES, Blowfish)

Як працює:
Birthday paradox attack на малі блоки
Після 2^32 блоків (32GB даних) → collision → витягує plaintext

Захист:
✅ Вимкнути 3DES
✅ Використовувати AES (128-bit блоки)

Статус: 3DES deprecated
```

#### 11. ROBOT (2017)

**Return Of Bleichenbacher's Oracle Threat**

```
Ціль: RSA key exchange

Padding oracle attack на RSA PKCS#1 v1.5

Може витягти приватний ключ або розшифрувати сесії

Захист:
✅ Використовувати ECDHE замість RSA key exchange
✅ Патчити вразливі реалізації

Статус: Виправлено у більшості реалізацій
```

#### 12. Raccoon Attack (2020)

```
Ціль: DH key exchange з reused DH keys

Timing side-channel attack

Дуже складна для експлуатації

Захист:
✅ TLS 1.3 (не вразливий)
✅ Патчити TLS 1.2 реалізації

Статус: Низький ризик, але патчи доступні
```

### Захист від атак - найкращі практики:

#### 1. Використовувати TLS 1.3

```nginx
# Nginx
ssl_protocols TLSv1.2 TLSv1.3;

# Або тільки TLS 1.3 (якщо сумісність дозволяє)
ssl_protocols TLSv1.3;
```

**Переваги TLS 1.3:**
- Видалено всі вразливі cipher suites
- PFS обов'язковий
- Зашифрований handshake
- Простіше → менше помилок

#### 2. Правильні Cipher Suites

```nginx
# Тільки безпечні cipher suites
ssl_ciphers 'ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256';

# Вимкнути
❌ RSA key exchange
❌ CBC mode
❌ 3DES
❌ RC4
❌ MD5, SHA1
```

#### 3. Perfect Forward Secrecy (PFS)

```
✅ Використовувати ECDHE або DHE
❌ Уникати RSA key exchange

Навіщо:
Якщо приватний ключ сервера скомпрометовано,
попередні сесії залишаються безпечними
```

#### 4. HSTS (HTTP Strict Transport Security)

```nginx
# Nginx
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
```

**Що робить:**
- Примушує браузер завжди використовувати HTTPS
- Захищає від SSL stripping attacks
- Захищає від downgrade attacks

**Параметри:**
- `max-age=31536000` — 1 рік
- `includeSubDomains` — для всіх субдоменів
- `preload` — для включення в HSTS preload list браузерів

#### 5. OCSP Stapling

```nginx
# Nginx
ssl_stapling on;
ssl_stapling_verify on;
ssl_trusted_certificate /path/to/chain.pem;
resolver 8.8.8.8 8.8.4.4 valid=300s;
resolver_timeout 5s;
```

**Переваги:**
- Приватність (CA не бачить, які сайти ви відвідуєте)
- Швидкість (немає додаткового запиту до CA)
- Надійність (працює навіть якщо OCSP сервер недоступний)

#### 6. Сильні DH параметри

```bash
# Генерація (займає час!)
openssl dhparam -out /etc/nginx/dhparam.pem 2048
```

```nginx
# Nginx
ssl_dhparam /etc/nginx/dhparam.pem;
```

**Навіщо:** Захист від Logjam атаки.

#### 7. Security Headers

```nginx
# Nginx
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
add_header X-Frame-Options "SAMEORIGIN" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-XSS-Protection "1; mode=block" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline' https://trusted.com" always;
```

## Найкращі практики

### Конфігурація сервера (2024):

#### Nginx - Оптимальна конфігурація:

```nginx
# SSL/TLS конфігурація
server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name example.com;

    # Сертифікати
    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # Протоколи (тільки TLS 1.2 та 1.3)
    ssl_protocols TLSv1.2 TLSv1.3;

    # Cipher suites (тільки безпечні, з PFS)
    ssl_ciphers 'ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256';
    ssl_prefer_server_ciphers off;

    # DH параметри (для DHE)
    ssl_dhparam /etc/nginx/dhparam.pem;

    # Session cache (для session resumption)
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;
    ssl_session_tickets off; # Або on для кращої продуктивності

    # OCSP Stapling
    ssl_stapling on;
    ssl_stapling_verify on;
    ssl_trusted_certificate /etc/letsencrypt/live/example.com/chain.pem;
    resolver 8.8.8.8 8.8.4.4 valid=300s;
    resolver_timeout 5s;

    # Security Headers
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # Решта конфігурації
    root /var/www/example.com;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }
}

# Редирект HTTP → HTTPS
server {
    listen 80;
    listen [::]:80;
    server_name example.com;
    return 301 https://$server_name$request_uri;
}
```

#### Apache - Оптимальна конфігурація:

```apache
<VirtualHost *:443>
    ServerName example.com
    DocumentRoot /var/www/example.com

    # SSL Engine
    SSLEngine on

    # Сертифікати
    SSLCertificateFile /etc/letsencrypt/live/example.com/cert.pem
    SSLCertificateKeyFile /etc/letsencrypt/live/example.com/privkey.pem
    SSLCertificateChainFile /etc/letsencrypt/live/example.com/chain.pem

    # Протоколи
    SSLProtocol -all +TLSv1.2 +TLSv1.3

    # Cipher suites
    SSLCipherSuite ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256
    SSLHonorCipherOrder off

    # OCSP Stapling
    SSLUseStapling on
    SSLStaplingCache "shmcb:logs/ssl_stapling(32768)"

    # HSTS
    Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"

    # Інші security headers
    Header always set X-Frame-Options "SAMEORIGIN"
    Header always set X-Content-Type-Options "nosniff"
    Header always set X-XSS-Protection "1; mode=block"
</VirtualHost>

# Редирект HTTP → HTTPS
<VirtualHost *:80>
    ServerName example.com
    Redirect permanent / https://example.com/
</VirtualHost>
```

### Висновок

TLS/SSL — критично важлива технологія для безпеки в Інтернеті. Сучасний стек:

```
✅ TLS 1.3 (або мінімум TLS 1.2)
✅ ECDHE для Perfect Forward Secrecy
✅ AEAD cipher suites (AES-GCM, ChaCha20-Poly1305)
✅ Сертифікати від Let's Encrypt або комерційних CA
✅ HSTS для примусового HTTPS
✅ Security headers
✅ Regular security audits
✅ Автоматичне продовження сертифікатів
```

**Ключові принципи:**
1. Використовувати найновіші версії протоколів
2. Видаляти застарілі та вразливі алгоритми
3. Perfect Forward Secrecy обов'язковий
4. Регулярні оновлення та моніторинг
5. Автоматизація (Certbot для Let's Encrypt)
