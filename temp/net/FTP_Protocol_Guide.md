## Вступ до FTP

### Що таке FTP?

**FTP (File Transfer Protocol)** — це стандартний мережевий протокол, що використовується для передачі файлів між клієнтом і сервером у комп'ютерній мережі. Це один з найстаріших протоколів, який досі активно використовується для завантаження та вивантаження файлів на веб-сервери, обміну даними та резервного копіювання.

**Простими словами:** FTP — це як файловий менеджер для інтернету. Він дозволяє вам підключатися до віддаленого комп'ютера (сервера), переглядати його файлову систему, завантажувати файли з нього та вивантажувати файли на нього.

### Ключові характеристики:

1.  **Клієнт-серверна архітектура:** FTP працює за моделлю, де клієнт ініціює з'єднання, а сервер відповідає на запити.
2.  **Використання двох з'єднань:** FTP використовує окремі канали для команд (Control Connection) і для передачі даних (Data Connection), що є його унікальною особливістю.
3.  **Stateful протокол:** На відміну від HTTP, FTP-сервер зберігає інформацію про стан клієнта (наприклад, поточну директорію) протягом сесії.
4.  **Підтримка аутентифікації:** Доступ до сервера зазвичай вимагає логіна та пароля. Існує також "анонімний" режим.
5.  **Текстовий і бінарний режими передачі:** FTP може передавати файли як звичайний текст або як бінарні дані.

## Історія та еволюція FTP

-   **1971:** Перша специфікація FTP (RFC 114) була опублікована Абхаєм Бхушаном. Це було задовго до появи TCP/IP. Протокол був простим і працював поверх NCP (Network Control Program).
-   **1980:** Виходить RFC 765, який адаптує FTP для роботи поверх TCP/IP. Це стало основою для сучасного протоколу.
-   **1985:** Публікується **RFC 959**, яка стала стандартом для FTP і залишається ним донині. Вона детально описує активний і пасивний режими, команди та коди відповідей.
-   **1997:** З'являється **FTPS (FTP over SSL)**, описаний в RFC 2228, як спроба додати шифрування до незахищеного протоколу FTP.
-   **1998:** Розроблено **SFTP (SSH File Transfer Protocol)** як частина протоколу SSH. Незважаючи на схожу назву, це абсолютно інший протокол, який не має нічого спільного з FTP.

Хоча сьогодні існують більш сучасні та безпечні альтернативи, FTP залишається популярним завдяки своїй простоті та широкій підтримці.

## Як працює FTP: Архітектура та режими

Ключова особливість FTP — це використання двох окремих TCP-з'єднань для керування сесією та передачі даних.

### Control Connection (Керуюче з'єднання)

-   **Порт:** Завжди встановлюється на **порт 21** сервера.
-   **Призначення:** Через це з'єднання клієнт надсилає команди (наприклад, `USER`, `PASS`, `LIST`, `RETR`), а сервер відповідає кодами статусу.
-   **Життєвий цикл:** Залишається відкритим протягом усієї сесії.

### Data Connection (З'єднання для даних)

-   **Порт:** Використовує динамічний порт.
-   **Призначення:** Через це з'єднання передаються власне файли та списки файлів (результат команди `LIST`).
-   **Життєвий цикл:** Створюється щоразу, коли потрібно передати дані, і закривається після завершення передачі.

Існує два режими встановлення цього з'єднання: активний і пасивний.

### Active Mode (Активний режим)

Це оригінальний режим роботи FTP.

**Як це працює:**

1.  Клієнт підключається з довільного порту (напр., `1025`) до **порту 21** сервера для встановлення керуючого з'єднання.
2.  Клієнт починає слухати інший довільний порт (напр., `1026`) для даних.
3.  Клієнт надсилає команду `PORT 192,168,1,10,4,2` (де `192.168.1.10` — IP клієнта, а `4*256+2=1026` — порт), повідомляючи серверу, на який порт підключатися.
4.  Сервер підключається зі свого **порту 20** до вказаного клієнтом порту (`1026`) для встановлення з'єднання для даних.
5.  Відбувається передача даних.

```
      Клієнт                                      Сервер
+------------------+                           +------------------+
| Port 1025 (Ctrl) | ----(1) CONNECT------>    | Port 21 (Ctrl)   |
|                  | <----(2) ACK----------    |                  |
|------------------|                           |------------------|
| Port 1026 (Data) | <----(4) CONNECT-------   | Port 20 (Data)   |
|                  |                           |                  |
+------------------+                           +------------------+
          ^
          |
  (3) PORT command (tells server to connect to 1026)
```

**Проблема:** Активний режим погано працює з сучасними фаєрволами та NAT на стороні клієнта. Фаєрвол клієнта блокує вхідні з'єднання від сервера на випадковий порт, оскільки ініціатором є сервер.

### Passive Mode (Пасивний режим)

Пасивний режим був створений для вирішення проблем активного режиму.

**Як це працює:**

1.  Клієнт підключається з довільного порту (напр., `1025`) до **порту 21** сервера для встановлення керуючого з'єднання.
2.  Клієнт надсилає команду `PASV`.
3.  Сервер починає слухати довільний порт (напр., `2048`) і надсилає клієнту відповідь, що містить його IP-адресу та цей порт.
4.  Клієнт підключається зі свого довільного порту (напр., `1026`) до вказаного сервером порту (`2048`) для встановлення з'єднання для даних.
5.  Відбувається передача даних.

```
      Клієнт                                      Сервер
+------------------+                           +------------------+
| Port 1025 (Ctrl) | ----(1) CONNECT------>    | Port 21 (Ctrl)   |
|                  |                           |                  |
|------------------|                           |------------------|
| Port 1026 (Data) | ----(4) CONNECT------>    | Port 2048 (Data) |
|                  |                           |                  |
+------------------+                           +------------------+
          |
 (2) PASV command
 (3) Server replies with port 2048
```

**Перевага:** У пасивному режимі **обидва з'єднання ініціюються клієнтом**, що дозволяє легко проходити через фаєрволи та NAT на стороні клієнта. **Сьогодні пасивний режим є стандартом де-факто.**

## Основні FTP-команди

Команди надсилаються через керуюче з'єднання у текстовому вигляді.

| Команда | Опис                                                             |
| :------ | :--------------------------------------------------------------- |
| `USER`  | Ім'я користувача для входу.                                      |
| `PASS`  | Пароль користувача.                                              |
| `PASV`  | Перехід у пасивний режим.                                        |
| `PORT`  | Перехід в активний режим і вказівка порту.                       |
| `LIST`  | Отримати список файлів і директорій у поточному каталозі.        |
| `RETR`  | Завантажити файл із сервера (Retrieve).                          |
| `STOR`  | Вивантажити файл на сервер (Store).                              |
| `DELE`  | Видалити файл на сервері.                                        |
| `CWD`   | Змінити робочу директорію (Change Working Directory).            |
| `PWD`   | Показати поточну робочу директорію (Print Working Directory).    |
| `MKD`   | Створити директорію на сервері (Make Directory).                 |
| `RMD`   | Видалити директорію на сервері.                                  |
| `QUIT`  | Завершити сесію.                                                 |
| `TYPE`  | Встановити режим передачі (`A` для ASCII, `I` для Image/Binary). |

**Приклад сесії:**

```
220 FTP Server ready.
USER myuser
331 Password required for myuser.
PASS mypassword
230 User myuser logged in.
PASV
227 Entering Passive Mode (192,168,1,1,19,132).
LIST
150 Opening data connection for directory listing.
226 Transfer complete.
RETR myfile.txt
150 Opening data connection for file transfer.
226 Transfer complete.
QUIT
221 Goodbye.
```

## Коди відповідей FTP

Сервер відповідає на кожну команду тризначним кодом.

-   **`1xx`**: Позитивна попередня відповідь. Команда прийнята, очікується наступна.
-   **`2xx`**: Позитивна фінальна відповідь. Команду успішно виконано.
-   **`3xx`**: Позитивна проміжна відповідь. Потрібні додаткові дані (напр., пароль).
-   **`4xx`**: Тимчасова негативна відповідь. Команда не виконана, але можна спробувати ще раз.
-   **`5xx`**: Постійна негативна відповідь. Команда не виконана.

| Код   | Значення                                     |
| :---- | :------------------------------------------- |
| `220` | Сервер готовий до роботи.                    |
| `230` | Користувач успішно увійшов.                  |
| `331` | Ім'я користувача коректне, потрібен пароль.  |
| `227` | Вхід у пасивний режим.                       |
| `150` | Статус файлу коректний, відкриття з'єднання. |
| `226` | Закриття з'єднання, передачу завершено.      |
| `530` | Помилка входу (неправильний логін/пароль).   |
| `550` | Файл не знайдено або доступ заборонено.      |
| `221` | Завершення сесії.                            |

## Безпека FTP: Проблеми та рішення

### Основні вразливості стандартного FTP:

1.  **Відсутність шифрування:** Логіни, паролі та всі дані передаються у відкритому вигляді. Будь-хто в мережі може їх перехопити (атака "Man-in-the-middle").
2.  **Вразливість до Brute-force атак:** Простий механізм аутентифікації дозволяє легко підбирати паролі.
3.  **Проблеми з фаєрволами:** Активний режим майже неможливо використовувати в сучасних мережах.

Через ці проблеми стандартний FTP вважається застарілим і небезпечним для використання в публічних мережах.

### FTPS (FTP over SSL/TLS)

**FTPS** — це розширення стандарту FTP, яке додає підтримку шифрування за допомогою протоколів SSL/TLS (тих самих, що використовуються в HTTPS).

**Існує два режими роботи FTPS:**

1.  **Implicit FTPS (Неявний):**

    -   Клієнт одразу встановлює зашифроване з'єднання на спеціальному порту (зазвичай **990**).
    -   Весь трафік, включаючи аутентифікацію, шифрується з самого початку.
    -   Вважається більш безпечним, але менш гнучким.

2.  **Explicit FTPS (Явний):**
    -   Клієнт підключається до стандартного **порту 21** і починає сесію як звичайний FTP.
    -   Клієнт надсилає команду `AUTH TLS` (або `AUTH SSL`), щоб "оновити" керуюче з'єднання до зашифрованого.
    -   З'єднання для даних також може бути зашифроване.
    -   Більш гнучкий і поширений варіант.

### SFTP (SSH File Transfer Protocol)

**SFTP** — це **не FTP**. Це абсолютно інший протокол, розроблений як частина SSH (Secure Shell).

-   **Архітектура:** Використовує **одне** зашифроване з'єднання (зазвичай на **порту 22**) для передачі команд і даних.
-   **Безпека:** Успадковує всі переваги SSH, включаючи надійне шифрування та аутентифікацію за допомогою ключів.
-   **Простота:** Відсутність проблем з активним/пасивним режимами робить його набагато простішим для роботи з фаєрволами.

### Порівняння: FTP vs. FTPS vs. SFTP

| Характеристика      | FTP (File Transfer Protocol)          | FTPS (FTP over SSL/TLS)                            | SFTP (SSH File Transfer Protocol)          |
| :------------------ | :------------------------------------ | :------------------------------------------------- | :----------------------------------------- |
| **Протокол-основа** | Власний                               | FTP                                                | SSH                                        |
| **Шифрування**      | ❌ Немає                              | ✅ Так (TLS/SSL)                                   | ✅ Так (SSH)                               |
| **Порти**           | 21 (команди), 20 (дані в акт. режимі) | 21 (explicit), 990 (implicit)                      | 22 (команди і дані)                        |
| **К-сть з'єднань**  | 2 (керуюче + для даних)               | 2 (керуюче + для даних)                            | 1 (все в одному)                           |
| **Проблеми з NAT**  | ✅ Так (особливо в активному режимі)  | ✅ Так (ті ж проблеми, що й у FTP)                 | ❌ Ні                                      |
| **Аутентифікація**  | Логін/пароль                          | Логін/пароль, сертифікати                          | Логін/пароль, SSH-ключі (більш безпечно)   |
| **Стандарт**        | RFC 959                               | RFC 4217                                           | Частина стандарту SSH (RFC 4251)           |
| **Висновок**        | 🔴 **Небезпечний**, застарілий        | 🟡 **Безпечний**, але складний через два з'єднання | 🟢 **Рекомендований**, безпечний і простий |

**Рекомендація:** Завжди віддавайте перевагу **SFTP**. Якщо SFTP недоступний, використовуйте **FTPS**. Уникайте використання звичайного FTP, особливо в незахищених мережах.

## Сценарії використання

-   **Веб-розробка:** Завантаження файлів сайту (HTML, CSS, JS, зображення) на хостинг.
-   **Резервне копіювання:** Автоматичне копіювання даних на віддалений сервер.
-   **Обмін великими файлами:** Передача файлів, які занадто великі для електронної пошти.
-   **Системне адміністрування:** Оновлення конфігураційних файлів на серверах.

## Найкращі практики

1.  **Використовуйте SFTP або FTPS:** Ніколи не використовуйте стандартний FTP для передачі чутливих даних.
2.  **Використовуйте пасивний режим:** Це стандарт де-факто, який уникає проблем з фаєрволами.
3.  **Створюйте надійні паролі:** Використовуйте довгі та складні паролі. Для SFTP надавайте перевагу аутентифікації за SSH-ключами.
4.  **Обмежуйте права користувачів:** Надавайте користувачам доступ лише до тих директорій, які їм потрібні.
5.  **Використовуйте бінарний режим:** Режим ASCII (`TYPE A`) може пошкодити файли, які не є простим текстом (наприклад, зображення або архіви). Бінарний режим (`TYPE I`) є універсальним і безпечним для будь-яких файлів.
6.  **Ведіть логи:** Налаштуйте сервер на ведення логів для моніторингу активності та виявлення підозрілих дій.

## Висновок

FTP — це один із стовпів інтернету, який відіграв ключову роль у його розвитку. Незважаючи на свій вік і проблеми з безпекою, він все ще знаходить своє застосування. Однак для сучасних завдань стандартний FTP є застарілим.

Розуміння його архітектури, особливо відмінностей між активним і пасивним режимами, є важливим для будь-якого мережевого інженера. А для безпечної передачі файлів сьогодні слід обирати його сучасні та захищені аналоги — **FTPS** і, в першу чергу, **SFTP**.
